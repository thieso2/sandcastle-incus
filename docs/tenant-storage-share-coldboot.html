Incident analysis

Tenant Storage Share — cold-boot autostart failure

Host big.thieso2.dev · observed 2026-06-01 · symptom: ping 10.248.1.20 timed out after a host reboot

Why a recipient machine refused to start, why it looked like a "dangling share", and how to make it impossible.

One-line summary. The shared directory was never deleted. A cross-tenant share is mounted as a raw host-path bind-mount into another tenant's legacy-mountpoint ZFS volume. Incus cannot see that dependency, so at cold boot it validates the recipient before the source volume is mounted — and refuses to start it.

1 · How a share is wired today

A Tenant Storage Share exposes a subdirectory of a source tenant's workspace to a recipient tenant's machine. The recipient gets a disk device pointing at a raw path on the host — with no pool= / volume reference that Incus could track.

Figure 1 — the recipient points at a raw path inside the source tenant's ZFS workspace. Incus sees a path, not a volume.
recipient source volume untracked dependency
Why mountpoint=legacy matters. A legacy-mountpoint ZFS dataset is not auto-mounted by ZFS at boot. Incus mounts it on demand — but only when an instance that properly references it (appears in the volume's used_by) starts. The recipient's raw bind-mount is invisible, so it never triggers that mount.

2 · What happens on a cold reboot

Everything works while the host has been running, because some source-side instance has already mounted the workspace dataset. A reboot removes that coincidence.

Figure 2 — a mount-ordering race: the recipient is validated before the (untracked) source volume is mounted.

Aggravating factors seen in this incident

  • crash incusd itself was SIGKILLed ~2s into the first boot (fixed by the kernel + incus 7.1 update), making boot ordering even more chaotic.
  • mirror The reverse share (sc-thieso2 → sc-some, dir docs) breaks default-test* the same way — so the source-side instances that would mount the volume can't start either. Deadlock-ish.
  • reconcile Reconcile drops any share whose source path is missing — so running it during a transient unmount would delete a valid share.

3 · Audit trail — what we could (not) reconstruct

We wanted to know when and by whom the share device was added. The answer: Incus keeps essentially nothing durable here.

Gap. There is no durable audit trail of share/volume/device mutations today. Fix: set loki.api.url + loki.types=lifecycle,logging so device adds, volume ops, and the requesting identity land in Loki with retention.

4 · Suggested solutions

Layered — a structural root fix plus several independent guards. Even one guard makes the cold-boot failure impossible.

Figure 3 — defense in depth. A is the structural cure; B–D are guards that stand on their own.

A · Root fix — track the dependency

Stop using raw host paths. Attach the source as a real Incus custom-volume reference so Incus mount-orders it before starting the recipient. Snag: shares expose a subdirectory of a workspace, which volume-attach doesn't natively subset — so this likely means a per-share child volume rather than a bind-mount into a shared dir. Highest leverage, largest change.

B · Boot ordering recommended first

  • A systemd unit ordered Before=incus-startup.service that mounts every workspace volume that is a share source; or
  • flip those volumes off mountpoint=legacy so ZFS mounts them at boot. Low effort, removes the race directly.

C · Fail-soft

  • Make the share mount non-fatal so a missing source never blocks the whole machine from booting.
  • Teach reconcile to distinguish "source volume unmounted" (transient → keep) from "grant revoked / dir deleted" (→ remove). Today it can't tell them apart — the dangerous part.

D · Integrity & observability

  • Cascade-reconcile all recipients when a source dir / volume / project / tenant is deleted (out-of-band deletions trigger nothing today).
  • Ship Incus lifecycle events to Loki and alert on instance autostart failures & start-validation errors (Prometheus already present).

5 · Recommendation

For durable protection at least risk: B (mount source volumes before incus-startup) + C (fix reconcile's transient-vs-deleted distinction) + D (enable Loki audit shipping). Pursue A as the longer-term share-feature redesign.

Immediate remediation already applied

  1. Saved the share device spec, removed the dangling device from io-dev, started it — reachable again.
  2. Updated kernel + Incus 7.1 → the incusd cold-boot SIGKILL is gone; a full reboot now recovers the whole host unattended.
  3. Verified end-to-end after reboot: sidecar Tailscale up, io-dev autostarts, ping 10.248.1.20 0% loss.
