# Base Images From Running Machines (`sc image`)

> **Status: ACCEPTED** (implemented; E2E-verified on obelix).

A tenant can turn a hand-customized machine into a reusable base image with
`sc image save <machine> <name>`, list saved bases with `sc image list`, remove
them with `sc image rm <name>`, and launch from one with the existing
`sc create <machine> --image <name>`. This is the "golden instance" path ŌĆö you
customize a live machine however you like, then snapshot it ŌĆö with **no image
build recipe** (no Dockerfile/buildah/registry).

## Decisions

1. **Snapshot-and-publish, non-disruptive.** `sc image save` creates a transient
   instance snapshot (`sc-save`), publishes *the snapshot* to a local image
   alias, then deletes the snapshot. The source machine is never stopped.

2. **The image is the rootfs only ŌĆö home and workspace are excluded for free.**
   A tenant's `/home` and `/workspace` are shared per-project custom volumes
   (`ensureV2ProjectVolumes`), attached as devices. `incus publish` captures only
   the instance rootfs, so a base image is purely the installed-software layer
   (packages, `/usr/local`, `/etc/skel`, system config) ŌĆö no source machine's
   files, no clearing step needed.

3. **Per-project, tenant-scoped aliases.** The image is published into the
   tenant's app Incus project (default `ŌĆ”-default`), which is exactly where
   `sc create --image` resolves a bare alias. `sc image list` surfaces only
   aliased images; re-saving a name replaces the prior image (idempotent).

4. **Generalization happens on the child's first boot, not at save time.** A
   machine cloned from a base must not inherit the source's SSH host keys,
   machine-id, or TLS leaf. Rather than editing the snapshot (which would mutate
   the running source), the boot cloud-init ŌĆö which re-runs per new instance ŌĆö
   gains a `sandcastle-generalize` step that runs *before* sshd: it removes the
   host keys (regenerated by `ssh-keygen -A`), removes `/etc/machine-id` so
   `systemd-machine-id-setup` mints a fresh one, and drops the stale leaf (the
   existing caddy-setup re-fetches a leaf and rewrites `machine.env`/Caddyfile for
   the new FQDN). Harmless no-op on a fresh stock machine; correctness lives in
   one place. Note cloud-init's own `ssh_deletekeys` already regenerates host
   keys per instance ŌĆö the generalize step makes host-key + machine-id freshening
   explicit rather than relying on that default, and is the sole source of the
   machine-id reset.

5. **No admin image-build pipeline.** This deliberately does *not* introduce an
   `sc-adm image bake` flow that renders a golden image from cloud-init. Saving a
   live machine is the whole feature; the tenant is the author.

## E2E result

Verified on obelix (`thieso2`, project `tc2-thieso2-default`): planted a rootfs
marker on `test`, `sc image save test mybase` published a 309.7 MB image
(re-save replaced it idempotently), `sc image list` showed it, and
`sc create probe --image mybase` launched in ~11 s (packages baked, no first-boot
install). `probe` carried the marker, got a **fresh** SSH host key and its own
`probe.default.obelix` FQDN + `probe.*` TLS leaf. The machine-id reset was
validated by running the corrected generalize body live (oldŌåÆnew random id).
