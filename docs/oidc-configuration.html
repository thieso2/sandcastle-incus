Sandcastle OIDC Configuration

How the Auth App is configured as the Sandcastle OIDC Provider, how it publishes discovery and JWKS, and how machines receive workload identity material for external cloud federation.

Summary Deployment Provider Machine Flow GCP Operations

Summary

Sandcastle has two separate identity paths. GitHub OAuth Login authenticates humans into the Auth App. The Sandcastle OIDC Provider issues short-lived workload identity tokens for machines. Do not treat GitHub OAuth as the OIDC issuer for cloud trust.

Admin config SANDCASTLE_AUTH_HOSTNAME, GitHub OAuth credentials, initial admins, database path.
Auth App Runs sandcastle-admin auth-app serve and owns OIDC issuer behavior.
OIDC metadata Serves discovery, JWKS, and workload token endpoints under the Auth Hostname.
Machine token Machine exchanges its runtime secret for a 15-minute RS256 JWT.
admin configuration -> Auth App sidecar -> public issuer endpoints -> external cloud trust

The issuer is the configured Auth Hostname. If the hostname has no scheme, Sandcastle normalizes it to https://auth.example.com.

Deployment Configuration

The Auth App can be started directly, or through the infrastructure sidecar plan. The direct command is:

sandcastle-admin auth-app serve \
  --listen :9444 \
  --database /var/lib/sandcastle/auth/auth.db \
  --auth-hostname auth.example.com \
  --github-client-id ... \
  --github-client-secret ... \
  --admin-github-users octocat,hubot

When managed by infrastructure creation, the plan writes these values into /etc/sandcastle/auth-app/env inside sc-auth-app and starts sandcastle-auth-app.service. Shared Caddy reserves the Auth Hostname and reverse-proxies both HTTP and HTTPS traffic to the Auth App on the infrastructure network.

OIDC Provider Endpoints

The Auth App registers the public OIDC endpoints and the internal workload token endpoint on the same HTTP handler:

EndpointBehavior
/t/<tenant>/.well-known/openid-configurationReturns the tenant-scoped issuer, JWKS URI, supported signing algorithm RS256, response type id_token, and public subject type.
/t/<tenant>/.well-known/jwks.jsonReturns active public signing keys for one tenant from the Auth Database. Private keys are never published.
/.well-known/openid-configuration and /.well-known/jwks.jsonRemain available for the legacy global issuer key set.
/internal/workload/tokenAccepts a machine runtime secret and returns a short-lived bearer token when the machine is enabled and still exists.

Signing Keys

Signing keys are bootstrapped lazily. The first discovery, JWKS, or token operation ensures an active RSA signing key exists. Sandcastle stores the public JWK and an encrypted private key in oidc_signing_keys. The encryption key is a 32-byte AES-GCM key stored in auth_app_meta under oidc_encryption_key.

Auth Database Tables

TableOIDC Role
oidc_signing_keysActive and retired signing keys. Stores kid, algorithm, encrypted private key, public JWK, activation time, and retirement state.
machine_runtime_secretsPer-machine runtime secret verifiers. The raw runtime secret is returned once and injected into the machine, but the database stores only the verifier.
cloud_identity_configsUser-owned cloud federation configuration. In v1, the implemented provider is GCP.

Machine Workload Identity Flow

1. Enable
Sandcastle generates a new per-machine runtime secret and stores only its verifier in machine_runtime_secrets.
2. Inject
The machine receives /var/lib/sandcastle/workload/runtime-secret and /var/lib/sandcastle/workload/token-endpoint.
3. Request
The machine posts tenant, project, machine, runtime_secret, and audience to /internal/workload/token.
4. Verify
The Auth App verifies the runtime secret, the enabled flag, the tenant, and the machine record.
5. Mint
The Auth App signs a 15-minute RS256 JWT with the active OIDC private key.

Token Claims

Minted workload identity tokens include:

  • iss: the per-tenant issuer URL.
  • sub: machine:<tenant>/<project>/<machine>.
  • aud: the Cloud Identity Audience requested by the machine.
  • iat, nbf, and exp: issue, valid-from, and expiry timestamps. The v1 token lifetime is 15 minutes.
  • tenant, project, and machine: Sandcastle workload identity vocabulary.
  • sandcastle_user_key and github_username: the user identity associated with the machine runtime secret.

GCP Cloud Identity Configs

Users can confirm or repair tenant-scoped cloud identity configs at:

https://auth.example.com/cloud-identities

For GCP, a config requires a Workload Identity audience. The subject token type defaults to urn:ietf:params:oauth:token-type:jwt when omitted. A service account impersonation URL is optional.

To configure the GCP side for one Sandcastle tenant, use the user CLI with the Auth Hostname and CLI Auth Token remembered by sc login and the active gcloud project, or pass --project explicitly:

sc cloud-identity gcp setup \
  --tenant acme

The command creates or updates the Workload Identity Pool and OIDC provider, sets the provider allowed audience to the //iam.googleapis.com/... provider resource name, creates the tenant service account if needed, grants roles/iam.workloadIdentityUser, saves the user-owned Sandcastle Cloud Identity Config named gcp for that tenant, and prints the audience and service account impersonation URL for confirmation. Project defaults set with sc project set-cloud-identity validate that the selected config exists for the current tenant before metadata is written.

Use the verification helper to display and check the GCP provider, service account IAM binding, and reachable Sandcastle issuer metadata:

scripts/verify-gcp-oidc.sh \
  --tenant acme \
  --auth-hostname auth.example.com

Operational Notes

  • The Auth Hostname is reserved infrastructure routing and should not be used as a user Public Route.
  • The JWKS endpoint publishes public keys only. Private signing keys remain encrypted in the Auth Database.
  • Machine runtime secrets are rotated when workload identity is enabled or re-enabled for a machine.
  • Cloud Identity Configs are user-owned and tenant-scoped. They can be selected per machine or set as a validated project default.
  • The sc cloud-identity gcp setup command configures tenant-scoped GCP trust and saves the tenant-scoped Cloud Identity Config when the CLI has a stored Auth Token. Users can still manage the saved config at /cloud-identities.
