1One fat binary, three faces

Everything is one executable built from cmd/sandcastle. The invocation name selects the role (main.go dispatches on argv[0]), and the same binary is what the admin deploy commands copy into every appliance — an appliance always runs its service from the exact build the operator installed.

InvocationTreeIncus credentialsNotes
sc / sandcastle User CLI (cli.Execute) Per-remote restricted cert — Execute points INCUS_CONF at the Sandcastle-managed incus config dir, so plain incus semantics apply to the enrolled remotes The current incus remote is the single source of truth: remote name sc-[<prefix>-]<tenant> encodes which install every command targets (installPrefixFromRemoteName)
sc-adm / sandcastle-admin Admin CLI (cli.ExecuteAdmin) Global ~/.config/incus admin certs / local unix socket — INCUS_CONF deliberately left alone Also hosts the long-running services as subcommands (auth-app serve, project broker-serve)
sc admin … Admin tree reached through the user binary as sc-adm Leading admin arg is stripped, then dispatches into ExecuteAdmin

Configuration resolves in a fixed order: CLI flags → SANDCASTLE_* env vars → the Infrastructure Seed File → built-in defaults. The user CLI additionally reads ~/.config/sandcastle/config.yml (tenant, remote, project, auth token, broker URL) — the durable half of a login session.

2Anatomy of an installation

One sc-adm install --prefix <p> creates one installation on an Incus host. Several installations coexist on one daemon — distinct prefixes give distinct appliance bridges, CIDR pools, project namespaces, and public hostnames; only the Incus daemon is shared (ADR-0015).

The install itself binds no host ports in tunnel mode: cloudflared runs inside the auth-app appliance and dials out. --ingress acme instead binds host :80/:443 with Let's Encrypt; --broker installs additionally deploy the project-broker appliance with a host port for the client-certificate tenant plane.

3The Incus object model

4The Auth App

A single Go HTTPS service (sc-adm auth-app serve, package internal/authapp) running inside the appliance, fronted by caddy + the install's public ingress. It owns identity (GitHub), the CLI Device Login, tenant provisioning, the token-gated tenant plane, workload identity (OIDC), and the DNS reconciler. State lives in a SQLite database inside the appliance (ADR-0002).

The CLI Device Login state machine

Three details of the poll handler are load-bearing. Detached provisioning: tenant bring-up (image pull, package install, tailscale join) takes minutes, and a poll arriving through a flaky Cloudflare edge can be cancelled long before that — so provisioning runs on a detached context.Background() with an 8-minute budget, serialized per device code by a mutex. Retry vs. deny: a transient provisioning error leaves the code approved and the next poll re-attempts; a tenant.TerminalProvisionError (immutable-suffix conflict, rejected suffix) flips it to denied so the client exits immediately with the message. Awaiting-tailnet polls: when the tenant supplied no auth key, the poll keeps re-ensuring provisioning and carries the sidecar's durable login.tailscale.com join URL until the sidecar gets a tailnet address.

CLI auth tokens & the SQLite database

A successful poll mints a CLI auth token (90-day TTL, stored as a SHA-256 verifier) — the bearer credential for /api/tenants, /api/projects, /api/shares* and /api/workload/enable. The database (PRAGMA foreign_keys=ON, dir mode 0700, additive migrations) holds:

Workload Identity (ADR-0005)

Each tenant gets its own OIDC issuer https://<auth-host>/t/<tenant> with a lazily generated RSA key (KID = SHA-256 of the public key). The /api/workload/enable endpoint mints a 32-byte Machine Runtime Secret, drops it into the machine at /var/lib/sandcastle/workload/, and stores only its verifier. The machine exchanges the secret at /internal/workload/token for a 15-minute JWT (sub=machine:<tenant>/<project>/<machine>, plus tenant/project/machine/github_username claims) — which a cloud's Workload Identity Federation (e.g. GCP, wired by sc cloud-identity gcp setup) accepts against the per-tenant issuer.

The V2 DNS reconciler (ADR-0018)

Because the appliance mounts the host socket, the serving auth-app also keeps every sidecar's DNS zone true to live Incus state. Triggers: an event subscription over GetEventsAllProjects filtered to instance-* actions (with settle re-passes at +3 s/+8 s to catch the DHCP lease that lands after the event), plus a 30-second sweep. Each pass is prefix-scoped (user.sandcastle.v2.prefix on the infra project) — one install's auth-app never touches another install's sidecars. It renders the zone from live machine IPs, compares against the live zone in the sidecar (a wiped sidecar self-heals), bumps the SOA serial monotonically (unix time), writes /etc/coredns/zones/db.<suffix> and reloads CoreDNS — records appear seconds after any incus launch, including freeform ones.

5Tenant provisioning, step by step

Every way a v2 tenant comes to exist — first CLI Device Login, sc-adm tenant create, or the broker admin plane — converges on one chain: EnsurePersonalTenant → PlanCreateV2 → CreateTenantV2 (internal/incusx/tenant_create_v2.go). Each step is idempotent; re-provisioning an existing tenant converges instead of failing.

Before the plan: what gets reused

  • ProvisionReuseInputs scans all managed projects and returns the tenant's own /24 + suffix (idempotent re-login keeps both) and every other tenant's CIDR as occupied. Scoping rules: a kind=infra project is "own" only when the owner matches and its v2.prefix matches this install; v1 kind=tenant projects are always occupied, never own — their bridge dnsmasq may still be live on the gateway IP.
  • CIDR allocation (internal/cidr): walk the pool in /24 steps, skip overlaps. Role octets inside the tenant /24: .1 gateway, .2 reserved (tailscale), .3 the sidecar/DNS address.
  • Suffix immutability: a requested --dns-suffix that contradicts the stored one is a TerminalProvisionError → the device login is denied, nothing half-provisioned.

CreateTenantV2 — the ordered mutation list

  1. Tenant bridge <prefix>-<tenant> (default project): ipv4.address=<.1>/24, NAT, dns.domain=<suffix>, and raw.dnsmasq=dhcp-option=6,<.3> — guests get the sidecar CoreDNS as resolver via DHCP, never the bridge dnsmasq (ADR-0018).
  2. Infra project <prefix>-<tenant> — the durable tenant record lives in its metadata: bridge, pool, suffix, CIDR, unix user, SSH key, install prefix.
  3. App project <prefix>-<tenant>-default — own image store (features.images=true), suffix metadata.
  4. Shared volumes workspace + home in the app project — initial.uid/gid=2000, mode 0775, security.shifted=true when the host supports idmapped mounts (Incus 7.x reports an empty kernel_features map, which counts as supported); without idmapped mounts the shared /home is omitted so VM sshd keeps working.
  5. Default profile in the app project — root disk, eth0 on the tenant bridge, both volume disks, and cloud-init.user-data creating the login user (UID 2000, sudo, the User SSH Public Key, openssh-server) with fqdn: {{hostname}}.<project>.<suffix>. The key is re-baked on every login — key rotation is just sc login --force.
  6. Sidecar profile + instance <prefix>-<tenant> in the infra project — stock images:debian/13 pulled on demand, NIC pinned to .3; an existing stopped sidecar is started, then waited to RUNNING.
  7. Sidecar software — CoreDNS release binary (arch-matched, not in apt), Tailscale from its official apt repo, a sandcastle-sidecar-network oneshot pinning the static .3, Corefile + zone + /etc/resolv.conf (symlink cleared first), CoreDNS unit, systemd-resolved masked.
  8. Tailnet join — tailscale up --advertise-routes=</24> --advertise-tags=tag:sandcastle --accept-dns=false. With an auth key: unattended, hard-gated on a tailnet IP. Without: a detached unit runs the interactive join and the login URL is carried back through awaiting-tailnet polls until the tenant approves. Then Incus Reach (ADR-0017): tailscale serve --bg --tcp=8443 tcp://<gateway>:8443 — raw TCP, so the host's own Incus certificate is what the client pins.
  9. Trust — the restricted certificate: either mint a certificate-add token (CreateToken, restricted to the tenant's projects) for first-time enrollment, or — shared client identity — EnsureClientCertificate unions this install's projects into the client's already-trusted certificate and the token goes unused.

6Networking & DNS

The tailnet path (ADR-0006 / ADR-0017)

Tenants bring their own tailnet; the server holds no Tailscale API key and the host itself is on no tailnet. The sidecar is the tenant's entire network presence: it advertises the tenant /24 as a subnet route (tagged tag:sandcastle, so an autoApprovers ACL rule approves it with zero touch) and proxies the host Incus API onto the tenant tailnet as a raw-TCP tailscale serve on :8443 — Incus Reach. The client's enrolled remote URL is https://<sidecar-tailnet-IP>:8443; because the proxy is raw TCP, the host daemon's own certificate is pinned end to end.

Subnet-to-subnet does not route. A machine that is merely resident in a subnet advertised by some other router can never reach the tenant /24 — the client must itself be a tailnet node with --accept-routes. sc login refuses to start the device flow otherwise.

Login routing verification — the five checks

  1. tailscale binary present;
  2. backend state Running (after tailscale set --accept-routes=true);
  3. accept-routes actually enabled;
  4. the tenant /24 is offered by a peer and a primary subnet router is elected (stale duplicate sidecar devices lose the election and blackhole the route — the check catches it);
  5. the real gate: TCP dial <gateway>:8443 and confirm the connection's local address is this machine's own CGNAT (100.64/10) address — proof the packet egressed via the tailnet and not a coincidentally-routable LAN with an overlapping /24.

DNS, server side: the sidecar CoreDNS (ADR-0018)

The tenant zone is the only authority under the suffix — no fallthrough to bridge dnsmasq (guest-asserted, single-label lease names cannot express the project). Record shapes: the canonical <machine>.<project>.<suffix> (+ wildcard) for every machine; the short alias <machine>.<suffix> only for default-project machines — never first-wins across projects. The .:53 catch-all serves recursion to tenant machines (it is their only resolver, handed out via the bridge's dhcp-option=6,<.3>) but answers REFUSED to tailnet sources (acl { block net 100.64.0.0/10 }) — an upstream-forwarded NXDOMAIN for another tenant's name would otherwise end a client's lookup and mask the answering server.

# /etc/coredns/Corefile (rendered per tenant)
castle:53 {
    errors
    file /etc/coredns/zones/db.castle castle
}
.:53 {
    errors
    acl { block net 100.64.0.0/10 }      # tailnet clients: REFUSED outside the zone
    forward . /etc/coredns/upstream-resolv.conf { force_tcp }
}

DNS, client side: one resolved link scope per suffix

Per-domain routing in systemd-resolved only works across link scopes — a global resolved.conf.d drop-in merges every tenant's server into one flat list where only the rotating "current server" is asked, so a second install kills the first zone. Each suffix therefore gets a persistent unit sandcastle-dns-<suffix>.service that creates a dummy link scdns-<hash>, gives it a deterministic 169.254/16 link-local address (resolved never activates an addressless link's scope), and pins DNS=<CoreDNS> Domains=~<suffix> to it. PartOf=systemd-resolved.service re-applies the runtime scope whenever resolved restarts. macOS needs none of this — /etc/resolver/<suffix> is natively per-domain.

7End-to-end flows

Flow: sc create web → running, named, reachable

Flow: sc project create backend (tenant plane)

  1. Client calls the auth-app POST /api/projects with the saved CLI auth token (tunnel installs), or the broker POST /v2/projects authenticated by the tenant's client certificate (broker installs). No flags needed — login recorded both.
  2. Server-side, over the host socket: create project <prefix>-<tenant>-backend, its home/workspace volumes, its default profile — then extend the caller's restricted certificate to cover the new project.
  3. Client writes the per-project remote <tenant>-backend into the shared incus config (pinned to the new Incus project) — sc create backend:api works immediately over the same certificate, no re-login.

Flow: sc-adm tenant delete <t> --yes --purge (v2)

  1. Scoped plan: only this install's projects (PlanDeleteV2); refuse without --purge (shared volumes live inside the app projects — no runtime-only subset).
  2. Per app project: force-stop + delete instances → detach volume devices from the default profile → delete home/workspace → delete images/profiles → delete project.
  3. Infra project: delete sidecar instance + profile → delete project. Then delete the tenant bridge.
  4. Left behind on purpose: the sidecar's tailnet device (BYO tailnet — remove in the Tailscale console or use ephemeral keys) and the client's resolver unit/remotes (stale until next login).
  5. A later login by the same user re-provisions from scratch — fresh /24 from the pool, suffix freely choosable again (the immutability record died with the infra project).

Flow: two installs, one client (shared identity)

  1. All enrollments share one incus config dir and one client keypair; each install is one remote (sc-<tenant>, sc-<prefix>-<tenant>), each pinned to its install's default project.
  2. Enrolling the second install can't redeem a token (the daemon already trusts the keypair) — login falls back to a cert-based incus remote add --auth-type=tls, and the server unions the new install's projects into the one trust entry.
  3. Every tenant lookup in the CLI is scoped to the install the current remote belongs to (installPrefixFromRemoteName) — a same-named tenant of the other install can never shadow yours in sc list, create, connect, status, project, or dns.
  4. DNS is per-suffix, not per-remote: both zones resolve simultaneously through their own link scopes; switching installs is just incus remote switch.

8Command reference — sc user

Every sc command runs with the tenant's restricted certificate (Execute forces INCUS_CONF to the Sandcastle-managed config dir), reaching Incus through the sidecar's Incus Reach. Machine references are [tenant/][project:]machine; the project defaults to the Current Project, then default. State chips: what the command creates, mutates, deletes, touches locally, or calls remotely.

9Command reference — sc-adm admin

The admin tree always talks to Incus with admin credentials (global ~/.config/incus or the local unix socket — inside appliances, the mounted host socket). sc admin … is the identical tree reached through the user binary. Every command takes --output text|json; destructive commands prompt unless --yes; anything with --dry-run renders its plan and touches nothing.
