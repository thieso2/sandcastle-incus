The mental model

Images flow through four places. Each arrow is a deliberate step — nothing is automatic end-to-end yet.

images/base, images/ai GHCR Incus host "big" (Dockerfiles in this repo) ghcr.io/<owner>/... ┌─────────────────────────────┐ │ ▲ │ default project │ │ build in sc-builder │ push (OCI) │ sandcastle/ai:latest │ ▼ appliance (podman) │ │ (OCI, from build) │ ┌──────────────────┐ ──────push──┘ │ │ convert+copy │ │ sc-builder │ ───────────────────import (OCI)────▶│ ▼ │ │ (rootless │ │ sc-<tenant> project │ │ podman build) │ ──podman export rootfs──┐ │ sandcastle/ai:latest │ └──────────────────┘ ▼ │ (system container) │ install-image-to-tenant │ │ │ re-imports as a SYSTEM │ ▼ │ CONTAINER & repoints ─────▶ tenant machines (sc create) └─────────────────────────────┘

Key facts

  • The build runs in an appliance (sc-builder, project sc-build) using rootless podman — not on your laptop. See ADR 0010.
  • The published artifact is an OCI image on GHCR. Incus runs OCI images as application containers (PID 1 = the entrypoint, no systemd) — that is not usable as a Sandcastle machine.
  • Machines need a system-container image (boots systemd as PID 1). That form is produced by exporting the built rootfs and re-importing it with type: container metadata.
  • Each tenant Incus project keeps its own copy of the image (sc-<tenant>:sandcastle/ai:latest). Updating the global default alias does not refresh tenant copies.

1. Build & publish the image admin

Build base + AI and publish to GHCR

Builds both images in the sc-builder appliance from the current working tree, pushes them to GHCR, and imports the OCI image into the default-project alias on the host.

SANDCASTLE_GHCR_TOKEN=`envpocket get SANDCASTLE_GHCR_TOKEN` mise run image:all:build-remote

Variants: image:base:build-remote and image:ai:build-remote build a single template. The AI variant resolves the latest codex/claude-code/gemini-cli versions automatically.

Tip: commit your changes first — a dirty tree publishes a <commit>-dirty tag. A clean tree publishes the plain <commit> tag and a reproducible image.

What it produces

  • ghcr.io/<owner>/sandcastle-base:<tag> and :latest (pushed)
  • ghcr.io/<owner>/sandcastle-ai:<tag> and :latest (pushed)
  • big:default aliases sandcastle/base:latest / sandcastle/ai:latest — OCI form, imported from GHCR

Verify the OCI images landed:

incus image list big: | grep sandcastle

Why publishing isn't enough important

After build-remote, the default alias points at an OCI image. If you launch it, Incus reports image.type: oci and runs the entrypoint (sleep infinity) as PID 1 — systemd never boots, so smbd, Caddy, and the configure scripts never run. A machine created from it comes up but is broken.

That is why there is an explicit install step: it converts the freshly built image into a system-container image and copies it into the tenant project. Until tenants can self-serve image pulls, the install step below is the supported path.

Do not point a tenant's sandcastle/ai:latest at the OCI image. Machines created from it will not boot systemd (no Samba, no Caddy, no networking heal).

2. Install the image into a tenant admin

One command

Run this after build-remote. It takes the image the appliance already built (podman local storage), exports the rootfs natively (no GHCR re-pull, no emulation), re-imports it as a system container, and repoints the tenant's alias — all on the host, so the multi-GB rootfs never crosses the network.

bash scripts/install-image-to-tenant.sh ai <tenant>

Examples:

# install the AI image into tenant "thieso2"
bash scripts/install-image-to-tenant.sh ai thieso2

# install the base image instead
bash scripts/install-image-to-tenant.sh base thieso2

What it does, step by step

  • podman export the built image's rootfs inside sc-builder → a tar file.
  • incus file pull the tar onto the host.
  • Wrap it in metadata.yaml with type: container and incus image import → a new fingerprint.
  • Repoint sc-<tenant>:sandcastle/ai:latest at the new system-container image.
  • By default also repoint default:sandcastle/ai:latest so future tenants get a bootable image. Set UPDATE_DEFAULT_ALIAS=0 to skip.

Useful environment overrides:

VariableDefaultPurpose
SANDCASTLE_IMAGE_SSH_HOSTbig.thieso2.devSSH host of the Incus server
SANDCASTLE_REMOTEbigincus remote name
GHCR_OWNERthieso2GHCR owner of the podman image
UPDATE_DEFAULT_ALIAS1also repoint the default-project alias

Verify the tenant got a system container

incus image list big: --project sc-<tenant> | grep ai:latest

The TYPE column must read CONTAINER and the description should say system container with a recent date.

3. Machines pick up the image user

New machines

The tenant alias only stamps a machine at create time, so a new image reaches a machine when you create it:

sc create <name>      # boots from the tenant's current sandcastle/ai:latest

Existing machines

Already-running machines keep the image they were created from. To move them onto a newer image, recreate them:

sc delete <name> --yes
sc create <name>
Persistence: /workspace and the user's home are backed by per-tenant volumes and survive recreate. Anything written elsewhere in the machine rootfs is lost on recreate.

Samba shares (AI/base image)

Each machine serves its home directory and /workspace over SMB on port 445. Share names are unique per machine so mounting several doesn't collide:

  • default project: <machine>-home, <machine>-workspace
  • other projects: <project>-<machine>-home, <project>-<machine>-workspace

Mount from macOS: Finder → ⌘K → smb://<machine>.<project>.<tenant>, user = the tenant Linux user. smbd autostarts but exposes nothing until a Samba account exists on the machine.

No Samba account is created for you. sc password set created it, and that command was removed with tenant v1 (issue #52): it addressed machines by the v1 instance name <project>-<machine> and so could never reach a v2 machine. Until a v2 replacement exists, create the account by hand on each machine: sc connect <machine> -- sudo smbpasswd -a $(id -un).

Troubleshooting

build-remote fails on import: unauthorized from ghcr.io

The GHCR package is private. The host pulls anonymously, so set the sandcastle-base and sandcastle-ai packages to public in GitHub package settings, then re-run.

A new machine has no Samba / no systemd (PID 1 is sleep infinity)

The tenant alias points at an OCI image. Re-run the install step (scripts/install-image-to-tenant.sh) so the tenant gets the system-container form, then recreate the machine.

A new machine boots the old image

The tenant project's cached copy wasn't refreshed (updating default alone does not update tenants). Run the install step for that tenant, then recreate the machine.

macOS “authentication error” mounting a share

Either no Samba account exists on that machine (create one with sc connect <machine> -- sudo smbpasswd -a $(id -un); sc password set was removed with tenant v1), or macOS cached a stale credential — delete the entry for the host in Keychain Access and reconnect. Avoid ! in passwords typed at an interactive zsh prompt (history expansion).

Roadmap: the install step is interim. The intended design is a global sc-images store plus a self-service sc image update that pulls the latest image into a tenant via the Auth App. Until then, scripts/install-image-to-tenant.sh is the command.
