Quickstart

Sandcastle admin and developer onboarding

A short operating guide for creating an Incus-backed Sandcastle tenant, granting a developer restricted access, and getting that developer productive in a lightweight project on their own machine.

For the complete Auth App, OIDC, and workload identity setup path, open Full Auth App Setup.

Install a Sandcastle (current)

A fresh Debian-based host goes from empty to a running Sandcastle in two commands. This supersedes the manual multi-step admin setup below (kept for reference and deep customization). Sandcastle requires no prebuilt images — everything is created from stock images:debian/13, pulled on demand — and no inbound ports unless you choose the ACME ingress.

# 1. install the latest Incus (Zabbly) + minimal init + incus-admin group
sudo sc-adm install-incus

# 2. install the whole server side (auth-app + broker) with public ingress.
#    --ingress cloudflare  → outbound tunnel, no inbound ports
#      (pass --cloudflare-tunnel-token for a dashboard-made tunnel, or
#       --cloudflare-api-token to have the installer create the tunnel + DNS)
#    --ingress acme        → host :80/:443 + Let's Encrypt (needs a public IP + an A record)
#    --ingress none        → bring your own edge
sc-adm install \
  --auth-hostname burg.example.dev \
  --github-client-id <id> --github-client-secret <secret> \
  --admin-github-users you \
  --ingress cloudflare --cloudflare-api-token <token>

The GitHub OAuth App's callback URL is https://<auth-hostname>/oauth/github/callback. --cidr-pool is optional (shared by auth-app + broker); keep pools distinct across installations that share a tailnet. --prefix (default sc) scopes an installation so several can share one host; the installer refuses to run when a same-prefix install already exists.

Then a user logs in from any Tailscale node:

sc login https://burg.example.dev            # --tailscale-auth-key to join the sidecar unattended,
                                             # or follow the printed tailnet join URL
sc c dev                                     # create + SSH into a machine as your login user
sc project create web --broker https://<tenant-gateway>:9443 --cert … --key …  # a second project

The client needs the incus client (apt-get install incus-client / brew install incus) and must be a Tailscale node — sc login refuses up front otherwise and prints the install one-liner. Full runbook: docs/e2e-sc2.md.

Keeping it updated

Updates are always operator/tenant-initiated, never automatic. When a new GitHub release is out:

sc-adm update --check     # fleet table: auth-app, broker, sidecars (stamped versions vs latest)
sc-adm update             # update the global components (auth-app + broker); idempotent
sc-adm update --prefix NAME   # pick the install when several share this Incus remote
sc-adm update --all-tenants   # operator override: also force-roll every tenant sidecar

One Incus remote can host several installs: sc-adm update picks the only one automatically, or names them and asks for --prefix when there is more than one. Tenants keep their own CLI and sidecar current with sc update (the CLI also prints a passive once-a-day notice when a release is available; Homebrew installs are told to brew upgrade sandcastle instead of self-replacing). The auth-app's web page shows a version card — amber with the update command when the appliance is behind. Rollback = sc-adm update --version v<older>.

Assumptions (legacy manual admin setup)

Admin machine

  • Has an unrestricted Incus remote configured.
  • Can build or import Sandcastle base and AI images.
  • Has the Sandcastle repo or installed CLI available.

Developer machine

  • Has the Incus client installed.
  • Can reach the admin-provided Incus remote over HTTPS.
  • Has Tailscale available for tenant private network access.

Use real values for placeholders such as REMOTE, alice, acme, website, and codex.website.acme. Run destructive commands only against disposable or intended resources.

Admin setup

Run these commands from an admin environment with broad Incus permissions.

1. Build the CLI

make build
./bin/sc version

# Optional Linux amd64 binaries for handoff:
mise run build:linux-amd64

2. Select the admin remote and image defaults

export SANDCASTLE_REMOTE=REMOTE
export SANDCASTLE_STORAGE_POOL=default
export SANDCASTLE_CIDR_POOL=10.248.0.0/16
export SANDCASTLE_INCUS_PROJECT_PREFIX=sc
export SANDCASTLE_BASE_IMAGE=sandcastle/base:latest
export SANDCASTLE_AI_IMAGE=sandcastle/ai:latest

3. Prepare images

Tenants use the base image for sidecars and the AI image as the default machine template.

# Rebuild base and AI images, upload both, then sync tenant-local aliases
mise run image:all:build-upload

To rebuild one image at a time:

mise run image:base:build-upload
mise run image:ai:build-upload

The AI task resolves the latest npm versions automatically. To pin versions:

SANDCASTLE_CODEX_VERSION=0.132.0 \
SANDCASTLE_CLAUDE_VERSION=2.1.145 \
SANDCASTLE_GEMINI_VERSION=0.42.0 \
  mise run image:ai:build-upload

Or import a pre-built image from a registry:

./bin/sc-adm image import ai SOURCE_REF

3a. Deploy the Auth App and broker (optional)

The manual, per-service infrastructure deploy that used to live here (sc-adm infra … plus a separate route broker) has been removed along with tenant v1. Server-side deployment is now a single command — see Install a Sandcastle (current) at the top of this page (sc-adm install), and Full Auth App Setup for the OIDC provider, workload identity, and Cloud Identity details. Public routes and the sc route command were removed with tenant v1 and have no replacement.

Dev/e2e without a GitHub OAuth app: install with sc-adm install --simulate-github-token <secret> and log in with sc login <auth-host> --simulate-token <secret> --as <username>. This approves the device login offline (no browser, no GitHub). Never enable simulated mode in production.

4. Create the tenant

./bin/sc-adm tenant create acme

This creates the tenant Incus project with an initial default lightweight project, tenant-local Sandcastle image aliases, tenant-private bridge network, isolated home/workspace/CA volumes, CoreDNS sidecar, and Tailscale sidecar. Both sidecars start immediately, but Tailscale is not yet activated — the container is running but tailscaled has not been started.

5. Create the developer certificate token

Generate the token on the admin host while SANDCASTLE_REMOTE points at the Incus server the developer can reach. The join token contains the server address, so the developer does not pass a separate host when adding the remote.

incus remote list
incus info REMOTE:
./bin/sc-adm user create alice --tenant acme

The --tenant acme flag embeds the tenant grant in the certificate token. When the developer accepts the token, their restricted certificate is created with access to acme immediately. If the token expires or is lost before use, issue a replacement with ./bin/sc-adm user token alice --tenant acme.

For a multi-tenant developer or local Tenant Storage Share test, keep tenant creation and access grant explicit: create the second tenant with ./bin/sc-adm tenant create skorfman, then grant the same User with ./bin/sc-adm tenant grant skorfman alice. The developer can then use sc tenant list and sc tenant switch skorfman to select the local Current Tenant.

Stop here for handoff. The developer can do the rest asynchronously after receiving the install instructions and pre-granted join token.

Admin handoff checklist

Send this once after creating the pre-granted token. The developer does not need a second admin callback before running tenant commands.

Give the developer Example
Remote add command sc remote add sandcastle-alice JOIN_TOKEN --tenant acme
Access grant The token was generated with --tenant acme, so accepting it creates a restricted certificate that can already access the tenant.
Tenant and project names Tenant acme; Alice can create lightweight projects such as website herself, then use machine refs such as website:codex.
Private DNS names Tenant suffix acme; machine hostnames use machine.project.tenant, for example codex.website.acme.
Expected trust warning Trusting the tenant CA lets that tenant mint private certificates trusted by the developer machine.

Verify later

After the developer accepts the token, the admin can list tenant users to confirm that Alice's certificate is present and scoped to acme.

./bin/sc-adm tenant users acme

Developer machine setup

Run these commands on the developer's own workstation.

1. Install the CLI locally

mise run install
sc version

This installs the user-facing sandcastle binary and sc alias into ~/.local/bin when that directory is already on PATH, otherwise into /usr/local/bin. Developers do not need sandcastle-admin or sc-adm; those are operator tools. If the admin provides prebuilt Linux amd64 binaries instead of a source checkout, install the user CLI and alias directly from the mise run build:linux-amd64 output.

2. Add the restricted remote

sc remote add sandcastle-alice JOIN_TOKEN_FROM_ADMIN --tenant acme

This accepts the join token, creates ~/.config/sandcastle/sandcastle-alice/incus/ with the restricted Incus TLS certs, and writes ~/.config/sandcastle/config.yml with the remote and tenant set. No env var exports needed — all subsequent sc commands pick up the config automatically.

3. Check project visibility

sc list
sc project list
sc status

4. Provide the developer SSH key

The self-service sc ssh-key command was removed with tenant v1. The key that authorizes Machine SSH Access is now supplied out of band: the developer passes it at login with sc login <auth-host> --ssh-public-key ~/.ssh/id_ed25519.pub, or the admin bakes it into the tenant's default project profile:

# Admin, on the admin host:
./bin/sc-adm tenant set-ssh-key acme "$(cat ~alice/.ssh/id_ed25519.pub)"

Sandcastle stores the key in tenant metadata and injects it into machines created or re-ensured after this point.

5. Create or select a lightweight project

sc project create website
sc config set project website
sc project status website

Docker is installed in the base machine image, but the daemon is disabled by default. Use sc project set-docker-autostart website on only for projects where new machines should start Docker automatically; this also enables nested container tooling for those machines. Otherwise developers can run sudo systemctl start docker inside an individual machine.

6. Attach the tenant to the developer's Tailscale network

sc tailscale up

The command runs tailscale up --timeout=60s inside the current tenant's Tailscale sidecar and advertises the tenant private CIDR. For unattended environments, pass an auth key with --auth-key tskey-auth-...; without one, Tailscale may print a browser auth URL but the command remains bounded by the timeout.

7. Local DNS resolver

The standalone sc dns setup command was removed. The local resolver for the tenant suffix is now installed automatically during sc login (pass --skip-setup to suppress it), pointing directly at the tenant CoreDNS sidecar. The remaining sc dns subcommands are teardown and uninstall, which remove that resolver config again.

8. Trust the tenant CA for private HTTPS

sc trust install

On macOS, the CLI writes trust to the user's login keychain with security add-trusted-cert. Override the keychain with SANDCASTLE_DARWIN_TRUST_KEYCHAIN when a different local trust target is needed. After a successful install, the CLI caches the tenant CA keypair at ~/.config/sandcastle/tenant-ca/<remote>/<tenant-project> and restores it into recreated tenant storage before checking local trust, which keeps one approved tenant CA stable across sc-adm tenant delete --purge and later recreates.

This is explicit local-machine trust. Only install trust for tenants the developer owns and understands.

9. Create and connect to a machine

sc create codex          # create only; does not attach
sc connect codex         # SSH into it (alias: sc c codex)
sc c codex               # create-if-missing, then SSH, in one step

Machines launch from images:debian/13/cloud by default. Override the image with --image <ref>, or pass --vm to launch a virtual machine instead of a container. sc create no longer attaches automatically (the old --detach/--background flags are deprecated no-ops); use sc connect to open a shell.

Connect refreshes Sandcastle's per-remote, per-tenant known-hosts file at ~/.config/sandcastle/<remote>/known_hosts/<tenant> before connecting, so reused names and addresses get the current SSH host key without sharing entries through the global ~/.ssh/known_hosts. Connect starts a stopped managed machine before keyscan and SSH, then uses the machine private IP. The tenant Tailscale sidecar advertises the tenant CIDR instead of giving each container its own Tailscale IP.

Verification

Post-setup checklist

After sc-adm tenant create and developer setup, verify each layer in order:

  1. Tenant resources are healthy — metadata, CIDR, DNS, network, volumes, and both sidecars are present and running.
  2. Tailscale is connected — the tenant private CIDR is advertised as an approved route in the tenant Tailnet.
  3. Local DNS resolves private hostnames — machine FQDNs like codex.website.acme resolve to private IPs.
  4. SSH connect works — sc c codex opens a shell without keyscan warnings or hangs.
  5. Tenant CA is trusted — curl https://codex.website.acme succeeds (no SSL error).
  6. Tenant CA matches the server cert — if curl fails after sc trust install, the keychain has a stale CA from a previous tenant creation. Run sc trust install again after rebuilding the CLI to force a fresh CA read from Incus.

Admin checks

# Tenant resource and sidecar health
sc-adm tenant status acme
sc-adm tenant list acme       # all resources across the tenant's projects

# Resources in the tenant infra project (sidecars + shared volumes);
# machines live in the per-project app projects sc-acme-<project>
incus list REMOTE: --project sc-acme
incus list REMOTE: --project sc-acme-website

# Verbose service logs (timestamps + durations for every request and work span)
journalctl -u sandcastle-auth-app -n 50 --no-pager
journalctl -u sandcastle-broker  -n 50 --no-pager
# Activity Log UI: as admin, open https://AUTH_HOSTNAME/logs — every user's
# activity is visible; a regular user's /logs shows only her own entries.

Developer checks

# Tenant status
sc status

# SSH connect (uses private IP, no Tailscale required)
sc connect codex
sc c codex

# HTTPS trust (must succeed after sc trust install)
curl https://codex.website.acme

Expected healthy shape

  • Tenant metadata check is ok.
  • Private CIDR and domain checks are ok.
  • Tenant network (sc-private) and volumes (sc-home, sc-workspace, sc-ca) are present.
  • sc-tailscale and sc-dns sidecars are running.
  • Tailscale route check is ok after authentication and route approval in the tenant Tailnet.
  • sc c <machine> connects immediately without SSH keyscan delays.
  • curl https://<machine>.<project>.<tenant> returns HTTP 200 or application response without SSL errors.

After tenant recreation

When a tenant is deleted and recreated, the following local state becomes stale and must be refreshed:

  • Known hosts — old host keys remain in the per-tenant known-hosts file (and in ~/.ssh/known_hosts). sc c removes and re-scans them automatically on first connect. SSH with StrictHostKeyChecking=accept-new also handles this without errors.
  • Tenant CA trust — the new tenant has a new CA key pair. Run sc trust install to install the fresh CA. The CLI always reads the CA directly from Incus (never from stale local cache) and installs it if the keychain has an older version.
  • Local DNS — DNS suffix does not change when a tenant is recreated with the same name; no action needed.

Disposable VM local e2e

Use the host-side VM harness for local DNS resolver and platform trust coverage. The harness creates a 20GiB root disk by default.

Cleanup and recovery

Developer cleanup

sc delete codex --yes
sc project delete website --yes
sc trust uninstall
sc dns teardown
sc tailscale down acme

sc delete codex asks for confirmation in an interactive terminal. Keep --yes in cleanup scripts and other non-interactive runs.

Admin cleanup

./bin/sc-adm tenant revoke acme alice
./bin/sc-adm user delete alice
./bin/sc-adm tenant delete acme
# For disposable tenants only:
./bin/sc-adm tenant delete acme --purge

--purge removes durable tenant state. Omit it when preserving home/workspace/CA volumes is required.
