Quickstart

Sandcastle admin and developer onboarding

A short operating guide for creating an Incus-backed Sandcastle tenant, granting a developer restricted access, and getting that developer productive in a lightweight project on their own machine.

Assumptions Admin setup Handoff Developer machine Verification Cleanup

Assumptions

Admin machine

  • Has an unrestricted Incus remote configured.
  • Can build or import Sandcastle base and AI images.
  • Has the Sandcastle repo or installed CLI available.

Developer machine

  • Has the Incus client installed.
  • Can reach the admin-provided Incus remote over HTTPS.
  • Has Tailscale available for tenant private network access.

Use real values for placeholders such as REMOTE, alice, acme, website, and codex.website.acme. Run destructive commands only against disposable or intended resources.

Admin setup

Run these commands from an admin environment with broad Incus permissions.

1. Build the CLI

make build
./bin/sc version

# Optional Linux amd64 binaries for handoff:
mise run build:linux-amd64

2. Select the admin remote and image defaults

export SANDCASTLE_REMOTE=REMOTE
export SANDCASTLE_STORAGE_POOL=default
export SANDCASTLE_CIDR_POOL=10.248.0.0/16
export SANDCASTLE_INCUS_PROJECT_PREFIX=sc
export SANDCASTLE_INFRA_PROJECT=sc-infra
export SANDCASTLE_BASE_IMAGE=sandcastle/base:latest
export SANDCASTLE_AI_IMAGE=sandcastle/ai:latest

3. Prepare images

Tenants use the base image for sidecars and the AI image as the default machine template.

# Build base image and upload (zstd-compressed, fast)
mise run image:base:build-upload

# Build AI image with latest Codex / Claude Code / Gemini CLI versions and upload
mise run image:ai:build-upload

The AI task resolves the latest npm versions automatically. To pin versions:

SANDCASTLE_CODEX_VERSION=0.132.0 \
SANDCASTLE_CLAUDE_VERSION=2.1.145 \
SANDCASTLE_GEMINI_VERSION=0.42.0 \
  mise run image:ai:build-upload

Or import a pre-built image from a registry:

./bin/sc-adm image import ai SOURCE_REF

3a. Create shared route infrastructure (optional)

Set the host Incus socket and admin binary when this environment will serve public route mutations through the broker sidecar.

export SANDCASTLE_ROUTE_BROKER_INCUS_SOCKET=/var/lib/incus/unix.socket
export SANDCASTLE_ADMIN_BIN=$PWD/bin/sc-adm
./bin/sc-adm infra create

The route broker sidecar runs sandcastle-admin route-broker serve and uses the mounted Incus socket to map mTLS client certificates to restricted Incus project grants.

4. Create the tenant

./bin/sc-adm tenant create acme

This creates the tenant Incus project with an initial default lightweight project, tenant-local Sandcastle image aliases, tenant-private bridge network, isolated home/workspace/CA volumes, CoreDNS sidecar, and Tailscale sidecar. Both sidecars start immediately, but Tailscale is not yet activated — the container is running but tailscaled has not been started.

5. Create the developer certificate token

Generate the token on the admin host while SANDCASTLE_REMOTE points at the Incus server the developer can reach. The join token contains the server address, so the developer does not pass a separate host when adding the remote.

incus remote list
incus info REMOTE:
./bin/sc-adm user create alice --tenant acme

The --tenant acme flag embeds the tenant grant in the certificate token. When the developer accepts the token, their restricted certificate is created with access to acme immediately. If the token expires or is lost before use, issue a replacement with ./bin/sc-adm user token alice --tenant acme.

Stop here for handoff. The developer can do the rest asynchronously after receiving the install instructions and pre-granted join token.

Admin handoff checklist

Send this once after creating the pre-granted token. The developer does not need a second admin callback before running tenant commands.

Give the developer Example
Remote add command sc remote add sandcastle-alice JOIN_TOKEN --tenant acme
Access grant The token was generated with --tenant acme, so accepting it creates a restricted certificate that can already access the tenant.
Tenant and project names Tenant acme; Alice can create lightweight projects such as website herself, then use machine refs such as website/codex.
Private DNS names Tenant suffix acme; machine hostnames use machine.project.tenant, for example codex.website.acme.
Expected trust warning Trusting the tenant CA lets that tenant mint private certificates trusted by the developer machine.

Verify later

After the developer accepts the token, the admin can list tenant users to confirm that Alice's certificate is present and scoped to acme.

./bin/sc-adm tenant users acme

Developer machine setup

Run these commands on the developer's own workstation.

1. Install the CLI locally

make build
make install
sc version

If the admin provides prebuilt Linux amd64 binaries instead of a source checkout, install the user CLI and alias directly from the mise run build:linux-amd64 output.

2. Add the restricted remote

sc remote add sandcastle-alice JOIN_TOKEN_FROM_ADMIN --tenant acme

This accepts the join token, creates ~/.config/sandcastle/sandcastle-alice/incus/ with the restricted Incus TLS certs, and writes ~/.config/sandcastle/config.yml with the remote and tenant set. No env var exports needed — all subsequent sc commands pick up the config automatically.

3. Check project visibility

sc list
sc project list
sc status

4. Publish the developer SSH key

sc ssh-key set --file ~/.ssh/id_ed25519.pub

The developer owns this key. Sandcastle stores it in tenant metadata and injects it into machines created or re-ensured after this point.

5. Create or select a lightweight project

sc project create website
sc config set project website
sc project status website

6. Attach the tenant to the developer's Tailscale network

sc tailscale up

The command runs tailscale up inside the current tenant's Tailscale sidecar and advertises the tenant private CIDR. For unattended environments, pass an auth key with --auth-key tskey-auth-....

7. Install local DNS forwarding

sc dns setup

The setup command applies tenant DNS records, starts the local DNS forwarder service, and installs the local resolver for the current tenant.

8. Trust the tenant CA for private HTTPS

sc trust install

On macOS, the CLI elevates the keychain write with sudo security when needed.

This is explicit local-machine trust. Only install trust for tenants the developer owns and understands.

9. Create and connect to a machine

sc create codex
# Automation-friendly:
sc create codex --detach

By default, Sandcastle uses the AI machine template. Use --template base for a smaller machine.

Machine creation refreshes the global ~/.ssh/known_hosts entries for the machine FQDN and private IP before connecting, so reused names and addresses get the current SSH host key.

Verification

Admin checks

./bin/sc-adm tenant status acme
./bin/sc-adm tenant users acme
incus project show REMOTE:sc-acme
incus list REMOTE: --project sc-acme

Developer checks

sc status
sc project status website
sc status codex
sc connect codex

Expected healthy shape

  • Tenant metadata check is ok.
  • Private CIDR and domain checks are ok.
  • Tenant network and volumes are present.
  • sc-tailscale and sc-dns sidecars are running.
  • Tailscale route check becomes ok after authentication and route approval.

For repo-level verification, the destructive scripts/e2e.sh incus tier covers tenant create/purge and listing, tenant DNS, the current sc create and sc connect machine flows, local trust, host override create/list/delete, infrastructure, and image-sync checks.

Disposable VM local e2e

Use the host-side VM harness for local DNS service, platform trust, and /etc/hosts mutation coverage. The harness starts a root systemd user service manager inside the VM so Linux systemctl --user paths are exercised.

Test tenant private DNS suffixes are derived from the disposable tenant names; there is no separate e2e domain-suffix setting.

scripts/e2e-local-vm.sh
SANDCASTLE_E2E_VM_KEEP=1 scripts/e2e-local-vm.sh

Cleanup and recovery

Developer cleanup

sc delete codex --yes
sc project delete website --yes
sc trust uninstall
sc dns teardown
sc tailscale down acme

Admin cleanup

./bin/sc-adm tenant revoke acme alice
./bin/sc-adm user delete alice
./bin/sc-adm tenant delete acme --yes
# For disposable tenants only:
./bin/sc-adm tenant delete acme --yes --purge

--purge removes durable tenant state. Omit it when preserving home/workspace/CA volumes is required.
