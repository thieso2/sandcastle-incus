Quickstart

Sandcastle admin and developer onboarding

A short operating guide for creating an Incus-backed Sandcastle project, granting a developer restricted access, and getting that developer productive on their own machine.

Assumptions Admin setup Handoff Developer machine Verification Cleanup

Assumptions

Admin machine

  • Has an unrestricted Incus remote configured.
  • Can build or import Sandcastle base and AI images.
  • Has the Sandcastle repo or installed CLI available.

Developer machine

  • Has the Incus client installed.
  • Can reach the admin-provided Incus remote over HTTPS.
  • Has Tailscale available for project private network access.

Use real values for placeholders such as REMOTE, alice, myproject, and myproject.project-tld. Run destructive commands only against disposable or intended resources.

Admin setup

Run these commands from an admin environment with broad Incus permissions.

1. Build the CLI

make build
./bin/sc version

2. Select the admin remote and project defaults

export SANDCASTLE_REMOTE=REMOTE
export SANDCASTLE_STORAGE_POOL=default
export SANDCASTLE_BASE_IMAGE=sandcastle/base:latest
export SANDCASTLE_AI_IMAGE=sandcastle/ai:latest

3. Prepare images

Projects use the base image for sidecars and the AI image as the default sandbox template.

# Build base image and upload (zstd-compressed, fast)
mise run image:base:build-upload

# Build AI image with latest Codex / Claude Code / Gemini CLI versions and upload
mise run image:ai:build-upload

The AI task resolves the latest npm versions automatically. To pin versions:

SANDCASTLE_CODEX_VERSION=0.132.0 \
SANDCASTLE_CLAUDE_VERSION=2.1.145 \
SANDCASTLE_GEMINI_VERSION=0.42.0 \
  mise run image:ai:build-upload

Or import a pre-built image from a registry:

./bin/sc admin image import ai SOURCE_REF

4. Create the project

./bin/sc admin project create alice/myproject \
  --domain myproject.project-tld

This creates the Incus project, project-local private bridge network, isolated home/workspace/CA volumes, CoreDNS sidecar, and Tailscale sidecar. Both sidecars start immediately, but Tailscale is not yet activated — the container is running but tailscaled has not been started.

4a. Activate Tailscale (optional)

Skip this if you don't need Tailnet access to the project's private network.

# Interactive (opens browser auth)
./bin/sc tailscale up alice/myproject

# Unattended with an auth key
./bin/sc tailscale up alice/myproject --auth-key tskey-auth-...

The sc-tailscale sidecar advertises the project's entire private CIDR (e.g. 10.248.0.0/24) as a subnet route. Once you approve the route in the Tailscale admin console, any Tailnet device can reach all project sandboxes directly by their private IP.

5. Create the developer certificate token

./bin/sc admin user create alice

Copy the printed join token into the handoff notes. The developer runs sc remote add with it. The admin must grant project access after the developer has added the remote.

6. Grant project access after the developer adds the remote

./bin/sc admin user grant alice alice/myproject

The grant updates the developer's restricted Incus certificate so normal commands can operate only inside allowed projects.

Admin handoff checklist

Give the developer Example
Remote add command sc remote add sandcastle-alice JOIN_TOKEN --owner alice
Project reference alice/myproject, or myproject when SANDCASTLE_OWNER is set.
Project domain myproject.project-tld
Expected trust warning Trusting the project CA lets that project mint certificates trusted by the developer machine.

Developer machine setup

Run these commands on the developer's own workstation.

1. Install or build the CLI

make build
# Optional:
make install

2. Add the restricted remote

sc remote add sandcastle-alice JOIN_TOKEN_FROM_ADMIN --owner alice

This creates ~/.config/sandcastle/sandcastle-alice/incus/ with the Incus TLS certs and writes ~/.config/sandcastle/config.yml with the remote and owner set. No env var exports needed — all subsequent sc commands pick up the config automatically.

3. Check project visibility

sc ls
sc status myproject

4. Attach the project to the developer's Tailscale network

sc tailscale up myproject

The command runs tailscale up inside the project Tailscale sidecar and advertises the project private CIDR. For unattended environments, use an auth key supplied through the supported CLI flag or environment flow.

5. Install local DNS forwarding

sc dns install myproject
sc dns refresh myproject

The local DNS forwarder routes project-domain lookups from the developer machine to the project DNS sidecar over the Tailscale-routed private CIDR.

6. Trust the project CA for private HTTPS

sc trust install myproject

This is explicit local-machine trust. Only install trust for projects the developer owns and understands.

7. Create and enter a sandbox

sc add myproject/codex
# Automation-friendly:
sc add myproject/codex --detach

By default, Sandcastle uses the project default template, currently ai. Use --template base for a smaller sandbox.

Verification

Admin checks

./bin/sc admin project status alice/myproject
incus project show REMOTE:sc-alice-myproject
incus list REMOTE: --project sc-alice-myproject

Developer checks

sc status myproject
sc inspect myproject/codex
sc enter myproject/codex

Expected healthy shape

  • Project metadata check is ok.
  • Private CIDR and domain checks are ok.
  • Project network and volumes are present.
  • sc-tailscale and sc-dns sidecars are running.
  • Tailscale route check becomes ok after authentication and route approval.

Cleanup and recovery

Developer cleanup

sc rm myproject/codex --yes
sc trust uninstall myproject
sc dns uninstall myproject
sc tailscale down myproject

Admin cleanup

./bin/sc admin project delete alice/myproject --yes
# For disposable projects only:
./bin/sc admin project delete alice/myproject --yes --purge

--purge removes durable project state. Omit it when preserving home/workspace/CA volumes is required.
