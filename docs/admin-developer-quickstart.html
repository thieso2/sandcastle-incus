Quickstart

Sandcastle admin and developer onboarding

A short operating guide for creating an Incus-backed Sandcastle tenant, granting a developer restricted access, and getting that developer productive in a lightweight project on their own machine.

For the complete Auth App, OIDC, and workload identity setup path, open Full Auth App Setup.

Assumptions Admin setup Handoff Developer machine Verification Cleanup

Assumptions

Admin machine

  • Has an unrestricted Incus remote configured.
  • Can build or import Sandcastle base and AI images.
  • Has the Sandcastle repo or installed CLI available.

Developer machine

  • Has the Incus client installed.
  • Can reach the admin-provided Incus remote over HTTPS.
  • Has Tailscale available for tenant private network access.

Use real values for placeholders such as REMOTE, alice, acme, website, and codex.website.acme. Run destructive commands only against disposable or intended resources.

Admin setup

Run these commands from an admin environment with broad Incus permissions.

1. Build the CLI

make build
./bin/sc version

# Optional Linux amd64 binaries for handoff:
mise run build:linux-amd64

2. Select the admin remote and image defaults

export SANDCASTLE_REMOTE=REMOTE
export SANDCASTLE_STORAGE_POOL=default
export SANDCASTLE_CIDR_POOL=10.248.0.0/16
export SANDCASTLE_INCUS_PROJECT_PREFIX=sc
export SANDCASTLE_INFRA_PROJECT=sc-infra
export SANDCASTLE_BASE_IMAGE=sandcastle/base:latest
export SANDCASTLE_AI_IMAGE=sandcastle/ai:latest

3. Prepare images

Tenants use the base image for sidecars and the AI image as the default machine template.

# Build base image and upload (zstd-compressed, fast)
mise run image:base:build-upload

# Build AI image with latest Codex / Claude Code / Gemini CLI versions and upload
mise run image:ai:build-upload

The AI task resolves the latest npm versions automatically. To pin versions:

SANDCASTLE_CODEX_VERSION=0.132.0 \
SANDCASTLE_CLAUDE_VERSION=2.1.145 \
SANDCASTLE_GEMINI_VERSION=0.42.0 \
  mise run image:ai:build-upload

Or import a pre-built image from a registry:

./bin/sc-adm image import ai SOURCE_REF

3a. Create shared route infrastructure (optional)

Set the Auth Hostname, GitHub OAuth credentials, initial Sandcastle Admin GitHub usernames, host Incus socket, and admin binary when this environment will serve the Auth App and public route mutations through infrastructure sidecars.

export SANDCASTLE_AUTH_HOSTNAME=auth.example.com
export SANDCASTLE_AUTH_GITHUB_CLIENT_ID=...
export SANDCASTLE_AUTH_GITHUB_CLIENT_SECRET=...
export SANDCASTLE_AUTH_ADMIN_GITHUB_USERS=octocat,hubot
export SANDCASTLE_AUTH_DEBUG_DEVICE_USER=octocat   # optional lab-only device approval bypass
export SANDCASTLE_AUTH_TAILSCALE_AUTHKEY=tskey-auth-...   # optional lab-only unattended Tailscale setup
export SANDCASTLE_ROUTE_BROKER_INCUS_SOCKET=/var/lib/incus/unix.socket
export SANDCASTLE_INFRA_TLS_MODE=internal   # optional debug TLS; default is acme
export SANDCASTLE_INFRA_CADDY_DATA_ARCHIVE=~/.config/sandcastle/infra-caddy-data/big/sc-infra/caddy-data.tgz   # optional ACME reuse
mise run build:linux-amd64
export SANDCASTLE_ADMIN_BIN=$PWD/bin/linux-amd64/sc-adm
./bin/sc-adm infra create

infra create and infra delete read exported environment variables and local .env defaults, then print a configuration banner before mutation. Verify the remote, infrastructure project, TLS mode, image aliases, uploaded Linux admin binary, Auth Hostname, GitHub OAuth client ID, initial admin users, and Incus socket path; the GitHub OAuth client secret is shown only as set or unset. infra delete removes infrastructure resources; add --purge only when deleting Sandcastle tenant projects and durable data for the configured project prefix.

The Auth App sidecar runs sandcastle-admin auth-app serve with a persistent SQLite database. Initial admins can sign in with GitHub, manage explicit GitHub users at /admin/allowlist, and manage Tenant Access at /admin/access. In disposable lab environments, SANDCASTLE_AUTH_DEBUG_DEVICE_USER enables POST /debug/device/approve to approve CLI Device Login as that allowlisted user without a browser session, and SANDCASTLE_AUTH_TAILSCALE_AUTHKEY lets approved CLI Device Login clients receive a Tailscale auth key for unattended tenant sidecar attachment. The route broker sidecar runs sandcastle-admin route-broker serve. Both sidecars use the mounted Incus socket when route infrastructure is configured this way.

The shared Caddy sidecar publishes host ports 80 and 443 through Incus proxy devices and routes the configured Auth Hostname to the Auth App over the infrastructure bridge network. With SANDCASTLE_INFRA_TLS_MODE=internal, Caddy uses a persistent internal CA cached at ~/.config/sandcastle/infra-ca/<remote>/<infra-project>, and infra create installs that CA into local trust after the sidecar starts. This lets one trusted debug CA survive infra delete --purge and later recreates. In ACME mode, export a known-good Caddy ACME archive with ./bin/sc-adm infra cert export; later infra create restores it from SANDCASTLE_INFRA_CADDY_DATA_ARCHIVE or the default ~/.config/sandcastle/infra-caddy-data/<remote>/<infra-project>/caddy-data.tgz before Caddy starts, reducing repeat Let's Encrypt issuance during e2e recreates. Reinstall or remove internal debug trust with ./bin/sc-adm infra trust install and ./bin/sc-adm infra trust uninstall.

The Auth App publishes OIDC discovery and JWKS endpoints under the Auth Hostname. Signing keys are bootstrapped into the SQLite database with encrypted private key storage.

Workload identity uses per-Machine runtime secrets. The Auth App stores only verifiers, mints 15-minute tokens, and lets users manage User-owned GCP Cloud Identity Configs at /cloud-identities.

Allowlisted users first see the Auth App Onboarding Page after GitHub OAuth. If they are not on the Login Allowlist, the page tells them to ask a Sandcastle Admin for access; browser-only GitHub login does not provision a Personal Tenant.

CLI Device Login starts with sc login https://auth.example.com. The CLI prints a verification URL and user code, prepares the User SSH Public Key for Machine SSH Access, and polls while the Auth App creates or ensures the user's Personal Tenant, Default Project, Tenant Tailnet readiness, and restricted Incus certificate add token. The CLI enrolls the Incus remote with its own local client key, then uses that newly enrolled Incus config to set up tenant DNS, install local tenant CA trust, and run tailscale up for a single accessible tenant.

Login Readiness requires the local Tailscale Prerequisite when a Tenant Tailnet is returned: the user must be connected to the selected Tenant Tailnet and have a Tailscale IP. Revoking Tenant Access or removing a user from the Login Allowlist also removes that user's managed Machine SSH Access block from Machines in the affected Tenant.

4. Create the tenant

./bin/sc-adm tenant create acme

This creates the tenant Incus project with an initial default lightweight project, tenant-local Sandcastle image aliases, tenant-private bridge network, isolated home/workspace/CA volumes, CoreDNS sidecar, and Tailscale sidecar. Both sidecars start immediately, but Tailscale is not yet activated — the container is running but tailscaled has not been started.

5. Create the developer certificate token

Generate the token on the admin host while SANDCASTLE_REMOTE points at the Incus server the developer can reach. The join token contains the server address, so the developer does not pass a separate host when adding the remote.

incus remote list
incus info REMOTE:
./bin/sc-adm user create alice --tenant acme

The --tenant acme flag embeds the tenant grant in the certificate token. When the developer accepts the token, their restricted certificate is created with access to acme immediately. If the token expires or is lost before use, issue a replacement with ./bin/sc-adm user token alice --tenant acme.

Stop here for handoff. The developer can do the rest asynchronously after receiving the install instructions and pre-granted join token.

Admin handoff checklist

Send this once after creating the pre-granted token. The developer does not need a second admin callback before running tenant commands.

Give the developer Example
Remote add command sc remote add sandcastle-alice JOIN_TOKEN --tenant acme
Access grant The token was generated with --tenant acme, so accepting it creates a restricted certificate that can already access the tenant.
Tenant and project names Tenant acme; Alice can create lightweight projects such as website herself, then use machine refs such as website/codex.
Private DNS names Tenant suffix acme; machine hostnames use machine.project.tenant, for example codex.website.acme.
Expected trust warning Trusting the tenant CA lets that tenant mint private certificates trusted by the developer machine.

Verify later

After the developer accepts the token, the admin can list tenant users to confirm that Alice's certificate is present and scoped to acme.

./bin/sc-adm tenant users acme

Developer machine setup

Run these commands on the developer's own workstation.

1. Install the CLI locally

make build
make install
sc version

If the admin provides prebuilt Linux amd64 binaries instead of a source checkout, install the user CLI and alias directly from the mise run build:linux-amd64 output.

2. Add the restricted remote

sc remote add sandcastle-alice JOIN_TOKEN_FROM_ADMIN --tenant acme

This accepts the join token, creates ~/.config/sandcastle/sandcastle-alice/incus/ with the restricted Incus TLS certs, and writes ~/.config/sandcastle/config.yml with the remote and tenant set. No env var exports needed — all subsequent sc commands pick up the config automatically.

3. Check project visibility

sc list
sc project list
sc status

4. Publish the developer SSH key

sc ssh-key set --file ~/.ssh/id_ed25519.pub

The developer owns this key. Sandcastle stores it in tenant metadata and injects it into machines created or re-ensured after this point.

5. Create or select a lightweight project

sc project create website
sc config set project website
sc project status website

6. Attach the tenant to the developer's Tailscale network

sc tailscale up

The command runs tailscale up --timeout=60s inside the current tenant's Tailscale sidecar and advertises the tenant private CIDR. For unattended environments, pass an auth key with --auth-key tskey-auth-...; without one, Tailscale may print a browser auth URL but the command remains bounded by the timeout.

7. Install local DNS

sc dns setup

The setup command applies tenant DNS records and installs a local resolver for the current tenant suffix. The resolver points directly at the tenant sc-dns sidecar.

8. Trust the tenant CA for private HTTPS

sc trust install

On macOS, the CLI writes trust to the user's login keychain with security add-trusted-cert. Override the keychain with SANDCASTLE_DARWIN_TRUST_KEYCHAIN when a different local trust target is needed. After a successful install, the CLI caches the tenant CA keypair at ~/.config/sandcastle/tenant-ca/<remote>/<tenant-project> and restores it into recreated tenant storage before checking local trust, which keeps one approved tenant CA stable across infra delete --purge and later recreates.

This is explicit local-machine trust. Only install trust for tenants the developer owns and understands.

9. Create and connect to a machine

sc create codex
# Automation-friendly:
sc create codex --detach

By default, Sandcastle uses the AI machine template. Use --template base for a smaller machine.

Machine creation refreshes the global ~/.ssh/known_hosts entries for the machine FQDN and private IP before connecting, so reused names and addresses get the current SSH host key. Connect uses a recorded Tailscale Machine IP when available and otherwise falls back to the private IP.

Verification

Admin checks

./bin/sc-adm tenant status acme
./bin/sc-adm tenant users acme
incus project show REMOTE:sc-acme
incus list REMOTE: --project sc-acme

Developer checks

sc status
sc project status website
sc status codex
sc connect codex

Expected healthy shape

  • Tenant metadata check is ok.
  • Private CIDR and domain checks are ok.
  • Tenant network and volumes are present.
  • sc-tailscale and sc-dns sidecars are running.
  • Tailscale route check becomes ok after authentication and route approval.

For repo-level verification, the destructive scripts/e2e.sh incus tier covers tenant create/purge and listing, tenant DNS, the current sc create and sc connect machine flows, local trust, host override create/list/delete, infrastructure, and image-sync checks.

Disposable VM local e2e

Use the host-side VM harness for local DNS resolver, platform trust, and /etc/hosts mutation coverage. The harness creates a 20GiB root disk by default.

Test tenant private DNS suffixes are derived from the disposable tenant names; there is no separate e2e domain-suffix setting.

scripts/e2e-local-vm.sh
SANDCASTLE_E2E_VM_KEEP=1 scripts/e2e-local-vm.sh

Run go test ./internal/e2e -run TestFirstRunLoginWithMockGitHubOAuth -count=1 -v to exercise the first-run login mock GitHub OAuth path without external GitHub calls.

Cleanup and recovery

Developer cleanup

sc delete codex --yes
sc project delete website --yes
sc trust uninstall
sc dns teardown
sc tailscale down acme

sc delete codex asks for confirmation in an interactive terminal. Keep --yes in cleanup scripts and other non-interactive runs.

Admin cleanup

./bin/sc-adm tenant revoke acme alice
./bin/sc-adm user delete alice
./bin/sc-adm tenant delete acme --yes
# For disposable tenants only:
./bin/sc-adm tenant delete acme --yes --purge

--purge removes durable tenant state. Omit it when preserving home/workspace/CA volumes is required.
