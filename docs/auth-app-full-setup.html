Sandcastle Full Auth App Setup

Step-by-step setup for an Incus-backed Sandcastle install with Auth App login, GitHub allowlisting, personal tenants, public route infrastructure, OIDC workload identity, and GCP external account credentials.

Prerequisites Environment Incus Images GitHub OAuth Infrastructure Admin UI CLI Login Workload Identity Public Routes Verify

1. Install Prerequisites

Run these on the admin host where you can manage the Incus server.

  • Go matching this repository's go.mod.
  • Incus client configured with an admin remote.
  • Docker, for local image builds.
  • Caddy-compatible public DNS if you plan to run public routes or the public Auth Hostname.
  • A GitHub OAuth App for Auth App login.
go version
incus version
docker version
gh auth status
Docker group membership may require a new shell session after installation. If docker ps works without sudo, no re-login is needed for Docker in that shell.

2. Create and Review .env

Start from the default template, then fill in values for your machine and DNS setup.

cp .env.default .env
$EDITOR .env
set -a
source .env
set +a

The CLI also reads local .env defaults from the current directory. Exported variables still take precedence.

Do not commit .env. It can contain GitHub OAuth secrets, Tailscale auth keys, public route account details, and paths to Caddy ACME archives. Caddy ACME archives contain private TLS key material and must be stored like secrets.

3. Configure Incus Access

The admin binary uses the global Incus config. The user CLI stores restricted remote credentials under ~/.config/sandcastle/<remote>/incus/.

incus remote list
incus project list "$SANDCASTLE_ADMIN_REMOTE:"

If the Auth App and Route Broker run inside infrastructure sidecars, mount the host Incus socket with SANDCASTLE_ROUTE_BROKER_INCUS_SOCKET. Both sidecars use that socket for trusted server-side mutations.

4. Build and Import Base Images

Build the base and AI images, then import them into Incus under the aliases used by tenant creation.

go build -o bin/sc ./cmd/sc
go build -o bin/sc-adm ./cmd/sandcastle-admin

SANDCASTLE_E2E_IMAGE_BUILD=1 \
SANDCASTLE_E2E_CODEX_VERSION=... \
SANDCASTLE_E2E_CLAUDE_CODE_VERSION=... \
SANDCASTLE_E2E_GEMINI_CLI_VERSION=... \
scripts/e2e.sh images

For a manual production image workflow, build images with your pinned tool versions and import them as SANDCASTLE_BASE_IMAGE and SANDCASTLE_AI_IMAGE.

5. Create the GitHub OAuth App

  1. Create a GitHub OAuth App owned by the organization or admin account.
  2. Set the callback URL to https://AUTH_HOSTNAME/oauth/github/callback.
  3. Copy the client ID and secret into SANDCASTLE_AUTH_GITHUB_CLIENT_ID and SANDCASTLE_AUTH_GITHUB_CLIENT_SECRET.
  4. Add the initial admin GitHub usernames to SANDCASTLE_AUTH_ADMIN_GITHUB_USERS.
The Auth App uses GitHub usernames as v1 identity keys and stores GitHub numeric account IDs as metadata. GitHub email is stored only as inactive metadata.

6. Create Shared Infrastructure

This creates the infrastructure Incus project, Caddy instance, Route Broker, and Auth App sidecar.

mise run build:linux-amd64
export SANDCASTLE_ADMIN_BIN=$PWD/bin/linux-amd64/sc-adm
sandcastle-admin infra create

# Preview only:
sandcastle-admin infra create --dry-run

The Auth App serves on the internal infrastructure network and shared Caddy routes SANDCASTLE_AUTH_HOSTNAME to it. The Auth App database is stored at /var/lib/sandcastle/auth/auth.db inside the Auth App sidecar. In disposable lab environments, SANDCASTLE_AUTH_DEBUG_DEVICE_USER enables POST /debug/device/approve to approve CLI Device Login as that allowlisted user without a browser session, and SANDCASTLE_AUTH_TAILSCALE_AUTHKEY lets approved CLI Device Login clients receive a Tailscale auth key for unattended tenant sidecar attachment. Use short-lived or disposable auth keys for this flow.

For debugging public TLS without Let's Encrypt, set SANDCASTLE_INFRA_TLS_MODE=internal before infra create. Caddy then serves certificates from a persistent internal CA cached at ~/.config/sandcastle/infra-ca/<remote>/<infra-project>, and infra create installs that CA into local trust after the sidecar starts. The same trusted debug CA is reused across infra delete --purge and later recreates. For production ACME e2e, run sandcastle-admin infra cert export after a successful issuance; later infra create restores that Caddy ACME data archive from SANDCASTLE_INFRA_CADDY_DATA_ARCHIVE or the default cache path before Caddy starts, so it can reuse a known-good Let's Encrypt cert and account. Reinstall or remove internal debug trust with sandcastle-admin infra trust install and sandcastle-admin infra trust uninstall. Tenant CA trust behaves similarly after first approval: the CLI caches trusted tenant CA keypairs at ~/.config/sandcastle/tenant-ca/<remote>/<tenant-project> and restores them into recreated tenants before checking local trust.

The public route tier needs real public DNS, an infrastructure host, and a Let's Encrypt email. Tailscale Funnel is not a drop-in replacement for the current wildcard DNS and Caddy/ACME public route model.

7. Bootstrap Admin UI

Open the Auth App hostname in a browser and sign in with a bootstrap admin GitHub username.

https://auth.example.com/login/github
https://auth.example.com/admin/allowlist
https://auth.example.com/admin/access
  1. Add explicit GitHub users at /admin/allowlist.
  2. Remove users there to block future GitHub and CLI login and revoke restricted Incus grants.
  3. Use /admin/access to view Users, Tenants, and tenant access grants.
  4. Grant or revoke Tenant Access from /admin/access when needed.

8. User CLI Login

Allowlisted users enroll their local CLI with browser-assisted Device Login.

sandcastle login https://auth.example.com

The CLI prints a verification URL and user code. After browser approval, the Auth App creates or ensures the user's Personal Tenant, ensures its Default Project, returns a restricted Incus certificate add token, and the CLI enrolls the remote with its own local private key.

The CLI sets the default tenant only when exactly one accessible Tenant is returned.

9. Configure Workload Identity

The Auth App publishes OIDC provider metadata under the Auth Hostname. See Sandcastle OIDC Configuration for the full provider and machine token flow.

curl https://auth.example.com/.well-known/openid-configuration
curl https://auth.example.com/.well-known/jwks.json

Users manage User-owned Cloud Identity Configs at /cloud-identities. For GCP, provide the Workload Identity audience and optional service account impersonation URL.

https://auth.example.com/cloud-identities

When workload identity is enabled for a Machine, Sandcastle rotates a Machine Runtime Secret, stores only a verifier in the Auth Database, injects token endpoint metadata, and writes GCP external account credentials plus shell environment variables into the Machine.

Machine FilePurpose
/var/lib/sandcastle/workload/runtime-secretMachine Runtime Secret.
/var/lib/sandcastle/workload/token-endpointAuth App internal token endpoint.
/var/lib/sandcastle/workload/gcp-credential.jsonGCP external account credential file.
/etc/profile.d/sandcastle-workload-identity.shExports GOOGLE_APPLICATION_CREDENTIALS and CLOUDSDK_AUTH_CREDENTIAL_FILE_OVERRIDE.

10. Public Routes

Public routes go through shared Caddy and the Route Broker. Configure these only when you have public DNS and an infrastructure host.

export SANDCASTLE_INFRA_HOST=203.0.113.10
export SANDCASTLE_LETSENCRYPT_EMAIL=ops@example.com
export SANDCASTLE_E2E_PUBLIC_DOMAIN=e2e.example.com

Leave SANDCASTLE_INFRA_TLS_MODE unset or set to acme for public ACME certificates. Use sandcastle-admin infra cert export after the first successful run to cache Caddy's ACME storage for later recreates.

User-created public routes cannot use the configured Auth Hostname; it is reserved for the Auth App.

11. Verify the Install

Start with unit and gated tests, then run destructive tiers only from an environment that intentionally sourced .env.

scripts/e2e.sh unit
scripts/e2e.sh gated
scripts/e2e.sh local

set -a
source .env
set +a
scripts/e2e.sh images
scripts/e2e.sh incus
scripts/e2e.sh restricted
scripts/e2e.sh tailscale
scripts/e2e.sh route-broker
TierValidates
unitAll Incus-free Go tests with a sanitized environment.
gatede2e compile and skip behavior with SANDCASTLE_E2E=0.
imagesDocker image build and Incus image import flow.
incusTenant, machine, DNS, route broker, local trust, and CLI create/connect flows.
restrictedRestricted client certificate token, grants, and machine lifecycle.
tailscaleTailscale routed access.
route-brokermTLS route broker mutations through infrastructure sidecars.
public-routesPublic Caddy/ACME route path; requires public DNS inputs.
local-vm mutates host resolver/trust/hosts state and is intended for a disposable VM. Run it only in that environment with SANDCASTLE_E2E_LOCAL_VM=1.
