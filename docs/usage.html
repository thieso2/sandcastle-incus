1. Environment setup

Build binaries

Build native binaries for local use, or cross-compile Linux amd64 binaries for handoff to Linux hosts.

mise run build
mise run build:linux-amd64  # writes bin/linux-amd64/{sandcastle,sc,sandcastle-admin,sc-adm}

Config file (recommended)

Store your defaults in ~/.config/sandcastle/config.yml so you don't need to set env vars on every call.

# ~/.config/sandcastle/config.yml
tenant: acme
project: website
remote: sc-alice

Once set, sc create codex works without any extra flags or exports.

Environment variables take precedence over the config file if both are set.

sc config set project website
sc config unset project
sc config show

Admin environment variables

Set these before running any sandcastle-admin command, or put them in a local .env file in the current directory. Exported environment variables override .env, and both override the config file for admin commands. Normal sandcastle (sc) user commands ignore local .env files and read only the saved Sandcastle config plus explicitly exported environment variables.

# Incus remote name (default: local)
export SANDCASTLE_REMOTE=big

# Storage pool name in Incus
export SANDCASTLE_STORAGE_POOL=default

# Private CIDR pool for tenant networks (optional; default 10.248.0.0/16)
export SANDCASTLE_CIDR_POOL=10.248.0.0/16

# Prefix for managed tenant Incus projects (optional; default sc)
export SANDCASTLE_INCUS_PROJECT_PREFIX=sc

# Infrastructure Incus project name (for route broker / shared Caddy)
export SANDCASTLE_INFRA_PROJECT=sc-infra

# Optional public Auth App hostname reserved by shared Caddy
export SANDCASTLE_AUTH_HOSTNAME=auth.example.com

# GitHub OAuth app credentials for Auth App login
export SANDCASTLE_AUTH_GITHUB_CLIENT_ID=...
export SANDCASTLE_AUTH_GITHUB_CLIENT_SECRET=...

# Initial Sandcastle Admin GitHub usernames, comma-separated
export SANDCASTLE_AUTH_ADMIN_GITHUB_USERS=octocat,hubot

# Optional lab-only device approval bypass user
export SANDCASTLE_AUTH_DEBUG_DEVICE_USER=octocat

# Optional lab/unattended Tailscale sidecar attachment key returned by Device Login
export SANDCASTLE_AUTH_TAILSCALE_AUTHKEY=tskey-auth-...

# Optional host Incus socket mount for Auth App provisioning and route broker mutation service
export SANDCASTLE_ROUTE_BROKER_INCUS_SOCKET=/var/lib/incus/unix.socket

# Optional prebuilt Linux admin binary uploaded into infrastructure sidecars
export SANDCASTLE_ADMIN_BIN=$PWD/bin/linux-amd64/sc-adm

# Let's Encrypt email for shared Caddy TLS
export SANDCASTLE_LETSENCRYPT_EMAIL=ops@example.com

# Optional exported Caddy ACME data archive restored during infra create
export SANDCASTLE_INFRA_CADDY_DATA_ARCHIVE=~/.config/sandcastle/infra-caddy-data/big/sc-infra/caddy-data.tgz

# Debug TLS mode: use Caddy's internal CA instead of public ACME
export SANDCASTLE_INFRA_TLS_MODE=internal

Incus certificate selection and admin_remote

sandcastle-admin (sc-adm) always uses the global Incus config (~/.config/incus/) which holds admin certificates. sandcastle (sc) user commands use the per-remote Sandcastle config directory (~/.config/sandcastle/<remote>/incus/) which holds the restricted user certificate. Local repo .env defaults are intentionally admin-only so they cannot redirect sc to the admin remote.

The admin remote is the Incus remote name as it appears in ~/.config/incus/config.yml (e.g. big). It may differ from the Sandcastle user remote (e.g. sc-alice). Configure it once with:

sc-adm config set admin_remote big
# or
export SANDCASTLE_ADMIN_REMOTE=big

When admin_remote is set, sc-adm uses it instead of remote. When unset, sc-adm auto-detects the admin remote by comparing server TLS certificates between the global and per-remote Incus configs.

Enable verbose output

Set VERBOSE=1 to see the Incus config directory and remote being used, per-operation detail from executors, and Incus API calls with durations.

export VERBOSE=1
sandcastle-admin tenant create acme
# [verbose] incus config: ~/.config/incus (default)
# [verbose] incus remote: big

2. Create & manage tenants admin

List tenants

sandcastle-admin tenant list
sandcastle-admin tenant list --json  # JSON output

Create a tenant

Provisions a tenant Incus project with private bridge network, storage volumes, Tailscale sidecar, CoreDNS sidecar, and tenant-local Sandcastle image aliases.

sandcastle-admin tenant create acme
Note: The tenant name becomes the private DNS suffix for machines in that tenant (e.g. codex.default.acme).
# Preview what will be created without mutating anything
sandcastle-admin tenant create acme --dry-run

Set or update tenant SSH key

The developer publishes her own SSH public key after accepting the tenant token. The key is injected into machines created or re-ensured after this point.

sandcastle ssh-key set --file ~/.ssh/id_ed25519.pub
sandcastle ssh-key set "ssh-ed25519 AAAA..."

Tenant status

sandcastle-admin tenant status acme

Delete a tenant (keep volumes)

Stops runtime resources (sidecars, network) but preserves home/workspace volumes.

sandcastle-admin tenant delete acme --yes

Purge a tenant (destroy everything)

sandcastle-admin tenant delete acme --yes --purge

3. Create users & credentials admin

Create a restricted user certificate

Issues a restricted certificate add-token for alice. Use --tenant to embed tenant access in the token so the user can onboard asynchronously.

sandcastle-admin user create alice --tenant acme

Example output:

User: alice
Certificate: sc-alice
Remote: sc-alice
Token: 7f3a...
Bootstrap:
  sc remote add sc-alice <join-token> --tenant acme
Share the bootstrap command with the user. They run sc remote add on their machine — this registers the remote, saves it as the default in ~/.config/sandcastle/config.yml, and applies the tenant grant embedded in the token.

Grant a user access to a tenant

Tenant access is managed with the tenant commands. Grants add the tenant's Incus project to the user's certificate restriction list.

sandcastle-admin tenant grant acme alice
sandcastle-admin tenant revoke acme alice
sandcastle-admin tenant users acme

Issue a new add-token for an existing user

Use this if the user lost their token or you need to re-register the remote. Include --tenant when the replacement token should be pre-granted.

sandcastle-admin user token alice --tenant acme

Delete a restricted user certificate

Removes the user's restricted Incus certificate entirely. This revokes all tenant access for that certificate.

sandcastle-admin user delete alice

CLI Device Login

Starts browser-assisted sign-in against the Auth App. The CLI prints a verification URL and user code, then polls until the browser approval completes, expires, or is denied.

sandcastle login https://auth.example.com

Login prepares the User SSH Public Key used for Machine SSH Access before starting the device flow. Pass --ssh-public-key <path> to choose a public key explicitly. Without that flag, the CLI uses ~/.ssh/id_ed25519.pub when it exists, otherwise it creates a local Sandcastle SSH Key at ~/.ssh/sandcastle_ed25519. The private key stays local. On approval, the Auth App stores the public key, reconciles it onto existing Personal Tenant machines, and writes it to Personal Tenant metadata so future machines receive the key during bootstrap.

sandcastle login --ssh-public-key ~/.ssh/id_ed25519.pub https://auth.example.com

After browser approval, the Auth App creates or ensures the user's Personal Tenant, ensures its Default Project, returns a restricted Incus certificate add token, and streams provisioning status through CLI polling. The CLI enrolls the remote with its own local client key and sets the default tenant only when exactly one accessible Tenant is returned.

For a single accessible tenant, CLI login enrolls the restricted Incus remote, applies tenant DNS, installs local resolver config, installs the tenant CA into local trust, and runs tailscale up in the tenant Tailscale sidecar. These post-login steps use the newly enrolled Incus config.

When the CLI Login Result includes a Tenant Tailnet, login verifies the local Tailscale client is connected to that selected Tenant Tailnet and has a Tailscale IP before reporting Login Readiness. Multi-tenant login does not silently join every accessible Tenant Tailnet.

Set VERBOSE=1 to print login step timings for SSH key preparation, device start and poll requests, Incus remote enrollment, DNS, trust, Tailscale, and local tailnet verification.

Login Readiness means the Incus credential, User SSH Public Key, Personal Tenant, Default Project, Tenant Tailnet, local DNS, local tenant CA trust, local Tailscale Prerequisite, and local Sandcastle config are ready. A successful first-run login prints the next command, currently sandcastle create dev.

User bootstrap (on the user's machine)

Run these commands after receiving the bootstrap output from admin.

# One command sets up the remote and writes tenant + remote to config
sc remote add sc-acme <join-token-from-admin> --tenant acme

# Verify — no env vars needed
sc list
One-time setup. sc remote add creates ~/.config/sandcastle/sc-acme/incus/ with Incus TLS certs and saves tenant + remote to ~/.config/sandcastle/config.yml. All subsequent sc commands pick up the config automatically.

Use incus directly against your remote

sc incus wraps the incus CLI with the correct INCUS_CONF and defaults to the current tenant's Incus project. Pass an explicit --project to override the project.

sc incus ls                          # list instances in your tenant
sc incus project list               # list Incus projects you can access
sc incus exec website-codex -- bash

4. Create & manage machines

Manage lightweight projects user

Projects are namespaces inside the current tenant. They do not create separate Incus projects, and restricted users can manage them after the tenant token is accepted.

sandcastle project list
sandcastle project create website
sandcastle project status website
sandcastle project delete website --yes

Create and connect to a machine user

Creates an Incus container machine from the selected template and connects to a shell.

Creation records the Machine's Tailscale IP when one is available. If the Tenant Tailnet is not attached yet, Sandcastle still finishes creation and connects through the machine's private IP.

sandcastle create website/codex

Create in background user

sandcastle create website/codex --detach

Choose a template user

Use ai for an AI-tool-equipped image, base for the minimal image.

sandcastle create website/ai-box --template ai

Enable nested container tools (Docker/Incus inside machine) user

sandcastle create website/builder --container-tools

Set a shared home subdirectory user

Multiple machines can share the same home-volume subdirectory (requires --share-home when a conflicting machine already exists).

sandcastle create website/box1 --home-dir shared
sandcastle create website/box2 --home-dir shared --share-home

Expose an app port user

Configures Caddy inside the machine to reverse-proxy port 3000 at the machine's private URL.

sandcastle create website/webapp --app-port 3000

Change app port on an existing machine user

sandcastle port set website/webapp 8080

Preview the creation plan without creating user

sandcastle create website/codex --dry-run

Connect to a running machine user

sandcastle connect website/codex
sandcastle c website/codex

# Run a specific command
sandcastle connect website/codex -- bash -c "echo hello"

connect uses SSH to the recorded Tailscale Machine IP when present, otherwise it falls back to the machine's private IP. In both cases it uses the private machine hostname as the host-key alias. Local DNS is useful for private URLs, but it is not required for Machine SSH Access.

If the machine is missing, connect logs that it is creating the machine, then connects after creation.

Lifecycle commands user

sandcastle start   website/codex
sandcastle stop    website/codex
sandcastle restart website/codex
sandcastle delete  website/codex --yes

delete prompts for confirmation when run interactively. Use --yes for scripts, pipes, and other non-interactive runs.

Show machine status user

sandcastle status website/codex
sandcastle status website/codex --json

List machines user

sandcastle list
sandcastle ls
sandcastle list --all-projects
sandcastle list --json

Text output is an aligned table with project, machine, FQDN, private IP, local creation time (yyyy-mm-dd hh:mm:ss), and state. Output includes unmanaged Incus instances. Machine FQDNs use <machine>.<project>.<tenant>, for example codex.website.acme.

5. Install local CA trust

Trust the tenant CA on your machine user

Makes the tenant's self-signed CA trusted by your OS/browser so machine HTTPS URLs work without warnings.

On macOS, the CLI installs trust into the user's login keychain with security add-trusted-cert. Override the keychain with SANDCASTLE_DARWIN_TRUST_KEYCHAIN when a different local trust target is needed. After a successful tenant CA trust install, Sandcastle caches that tenant CA keypair at ~/.config/sandcastle/tenant-ca/<remote>/<tenant-project> and restores it into a recreated tenant before checking local trust, so an approved tenant CA can survive infra delete --purge and later recreates.

sandcastle trust install
sandcastle trust install acme

Remove the CA from local trust user

sandcastle trust uninstall
sandcastle trust uninstall acme

Preview trust changes

sandcastle trust install   --dry-run
sandcastle trust uninstall --dry-run

6. Attach Tailscale

How it works

Every tenant gets an sc-tailscale sidecar container created automatically during tenant create. It sits on the tenant's private bridge at .2 (e.g. 10.248.0.2) and has a /dev/net/tun device.

The container is started but Tailscale is not activated yet — tailscaled and tailscale up are only run when you explicitly call tailscale up.

Once activated, the sidecar acts as a subnet router: it advertises the entire tenant private CIDR (e.g. 10.248.0.0/24) to your Tailnet. Any device on your Tailnet can then reach all machines in the tenant directly by their private IP, without needing Tailscale installed in each machine.

Tailscale admin console step: After running tailscale up, approve the advertised route in your Tailscale admin console under the machine's routing settings.

Activate Tailscale on a tenant (interactive)

Starts tailscaled inside the sidecar and runs tailscale up --timeout=60s --advertise-routes=<tenant-cidr>. Without an auth key, Tailscale may print a browser auth URL; the command is still bounded by the timeout so unattended flows fail cleanly.

sandcastle tailscale up
sandcastle tailscale up acme  # explicit tenant

Activate with an auth key (unattended)

sandcastle tailscale up --auth-key tskey-auth-...

Generate auth keys in the Tailscale admin console. Use a reusable key if you need to re-run this after a container restart.

Check Tailscale status

sandcastle tailscale status

Reports the Tailscale state, Tailnet name, assigned IPs, and which routes are being advertised.

Detach Tailscale

Runs tailscale down inside the sidecar. The container keeps running; only the Tailnet attachment is removed.

sandcastle tailscale down

7. Publish public routes

Create a public HTTP route user

Routes external HTTPS traffic for myapp.example.com to the machine's app port through the shared Caddy infrastructure.

# Requires SANDCASTLE_ROUTE_BROKER_URL to be set
sandcastle route create myapp.example.com website/webapp
DNS proof required: The route broker will tell you which DNS record to create before the route is activated.

List active routes user

sandcastle route list

Show a route user

sandcastle route status myapp.example.com

Delete a route user

sandcastle route delete myapp.example.com

8. Shared infrastructure admin

Create shared infrastructure

Provisions the shared Caddy reverse proxy, Auth App, and route broker inside a dedicated Incus project.

The route broker sidecar runs sandcastle-admin route-broker serve. When SANDCASTLE_ROUTE_BROKER_INCUS_SOCKET is set, the route broker and Auth App sidecars mount the host Incus socket and use the uploaded Linux sandcastle-admin binary from SANDCASTLE_ADMIN_BIN, falling back to SANDCASTLE_BIN. On macOS, build it with mise run build:linux-amd64 and use bin/linux-amd64/sc-adm; infra create rejects macOS binaries before mutation.

The Auth App sidecar runs sandcastle-admin auth-app serve with a persistent SQLite database at /var/lib/sandcastle/auth/auth.db. When SANDCASTLE_AUTH_HOSTNAME is set, shared Caddy reserves that hostname and routes it to the Auth App. CLI Device Login provisions Personal Tenants only after browser approval, not at allowlist add time or browser-only GitHub login time.

The shared Caddy sidecar publishes host ports 80 and 443 with Incus proxy devices and routes the configured Auth Hostname to the Auth App over the infrastructure bridge network. Set SANDCASTLE_INFRA_TLS_MODE=internal for debugging to have Caddy serve certificates from its internal CA instead of public ACME; infra create reuses a persistent debug CA from ~/.config/sandcastle/infra-ca/<remote>/<infra-project> and installs that CA into local trust after the sidecar starts. Override the cache location with SANDCASTLE_INFRA_CA_DIR. In ACME mode, infra create also restores an existing Caddy ACME data archive from SANDCASTLE_INFRA_CADDY_DATA_ARCHIVE, or from ~/.config/sandcastle/infra-caddy-data/<remote>/<infra-project>/caddy-data.tgz when that file exists, before restarting Caddy. This lets e2e recreates reuse a known-good Let's Encrypt certificate and account without ordering a new cert every run.

Initial Sandcastle Admins from SANDCASTLE_AUTH_ADMIN_GITHUB_USERS can sign in through GitHub and manage the Login Allowlist at /admin/allowlist and Tenant Access at /admin/access. The allowlist accepts explicit GitHub users only; password login is not available. For disposable lab integrations, SANDCASTLE_AUTH_DEBUG_DEVICE_USER enables POST /debug/device/approve to approve CLI Device Login as that allowlisted user without a browser session. SANDCASTLE_AUTH_TAILSCALE_AUTHKEY can be set during infra create to return a Tailscale auth key to approved CLI Device Login clients for unattended tenant sidecar attachment; this is intended for disposable integration environments and is not stored in Incus metadata.

infra create and infra delete print a configuration banner before mutation, including the selected remote, infrastructure project, project prefix, TLS mode, image aliases, uploaded admin binary, Auth Hostname, GitHub OAuth client ID, initial admin users, and Incus socket path. The GitHub OAuth client secret is redacted and shown only as set or unset.

The Auth App also serves public OIDC provider metadata at /.well-known/openid-configuration and public signing keys at /.well-known/jwks.json. The configured Auth Hostname is the issuer URL; private signing keys are stored encrypted in the Auth Database. See Sandcastle OIDC Configuration for the full provider and machine token flow.

Users can manage User-owned Cloud Identity Configs at /cloud-identities. Machines that opt into workload identity receive a rotated runtime secret, Auth App token endpoint metadata, and, when a GCP config is selected, an external account credential file plus GOOGLE_APPLICATION_CREDENTIALS and CLOUDSDK_AUTH_CREDENTIAL_FILE_OVERRIDE environment variables. Cloud Identity Configs are not applied automatically at Tenant or Project level.

sandcastle-admin infra create
sandcastle-admin infra create --dry-run   # preview only

Reuse a working ACME certificate

After a successful production ACME run, export Caddy's ACME storage from the running infrastructure. Later infra create calls in ACME mode restore that archive automatically before Caddy starts, so Caddy can serve the cached cert and keep normal renewal state.

The archive contains private TLS key material and ACME account state. Store it like a secret and do not commit it.

sandcastle-admin infra cert export
sandcastle-admin infra cert export --archive /secure/path/caddy-data.tgz

Trust infrastructure debug TLS locally

When using SANDCASTLE_INFRA_TLS_MODE=internal, infra create installs Caddy's persistent internal root CA locally after infrastructure creation. Because the CA is cached outside Incus, the same trusted root is reused across infra delete --purge and later recreates. Use these commands to reinstall or remove the same debug TLS trust manually.

sandcastle-admin infra trust install
sandcastle-admin infra trust uninstall

Delete shared infrastructure

Deletes the shared infrastructure runtime resources. Add --purge to also delete Sandcastle tenant projects and durable data for the configured project prefix.

sandcastle-admin infra delete
sandcastle-admin infra delete --purge

Serve the route broker

sandcastle-admin route-broker serve \
  --listen :9443 \
  --cert   /etc/caddy/certs/broker.crt \
  --key    /etc/caddy/certs/broker.key

Serve the Auth App

sandcastle-admin auth-app serve \
  --listen :9444 \
  --database /var/lib/sandcastle/auth/auth.db \
  --auth-hostname auth.example.com \
  --github-client-id ... \
  --github-client-secret ... \
  --admin-github-users octocat,hubot \
  --debug-device-user octocat

The debug device user flag is intended only for disposable integration environments. It registers POST /debug/device/approve, which approves a pending device code as the configured allowlisted user. Use --tailscale-auth-key only for unattended lab login flows where approved CLI clients should receive a Tailscale auth key for tenant sidecar attachment.

Manage the Login Allowlist

Open https://auth.example.com/admin/allowlist as a Sandcastle Admin. Adding a user verifies the GitHub username and stores the normalized username plus GitHub numeric account ID. Removing a user blocks future GitHub and CLI login, revokes their restricted Incus certificate grants, and removes their User SSH Public Key from Machines without deleting their Personal Tenant or Machines.

Manage Tenant Access

Open https://auth.example.com/admin/access as a Sandcastle Admin to view Users, Tenants, and restricted Incus certificate grants. Grant and revoke actions use the same project grant model as sandcastle-admin tenant grant and sandcastle-admin tenant revoke.

9. Manage images admin

Build and upload the AI image (latest tool versions) — recommended

Resolves the latest npm versions of Codex, Claude Code, and Gemini CLI at runtime, builds the OCI image, and imports it into Incus via zstd-compressed tarball.

export SANDCASTLE_REMOTE=big
mise run image:ai:build-upload

Pin specific versions by setting env vars before running:

SANDCASTLE_CODEX_VERSION=0.132.0 SANDCASTLE_CLAUDE_VERSION=2.1.145 SANDCASTLE_GEMINI_VERSION=0.42.0 \
  mise run image:ai:build-upload

Build and upload the base image

The base image includes tmux, vim, generated locales, and refreshed forwarded SSH agent sockets across reconnects.

export SANDCASTLE_REMOTE=big
mise run image:base:build-upload

All image mise tasks

Build an AI image manually (explicit versions)

sandcastle-admin image build ai \
  --codex-version  0.132.0 \
  --claude-version 2.1.145 \
  --gemini-version 0.42.0

Build the base image manually

sandcastle-admin image build base

Import an OCI image into Incus

sandcastle-admin image import base docker://ghcr.io/example/sandcastle-base:latest
sandcastle-admin image import ai   docker://ghcr.io/example/sandcastle-ai:latest

Sync an already-imported image to the alias

sandcastle-admin image sync sandcastle/base:debian-13

10. E2E scripts

Run the disposable VM local mutation tier

Creates a local Incus VM with a 20GiB root disk by default, installs Go, mise, and nested Incus, seeds nested image aliases from the host, starts root's systemd user service manager for systemctl --user coverage, copies the checkout, and runs the VM-only local DNS, trust, and hosts e2e tier inside the VM.

scripts/e2e-local-vm.sh
SANDCASTLE_E2E_VM_KEEP=1 scripts/e2e-local-vm.sh   # keep VM for diagnosis

Run the first-run login mock OAuth flow

The mock GitHub OAuth e2e test simulates GitHub authorization without network access, drives the Auth App onboarding and Device Login approval flow, uploads the User SSH Public Key, verifies Personal Tenant and Tenant Tailnet readiness in the CLI Login Result, and checks that connect planning can target the recorded Tailscale Machine IP when present.

go test ./internal/e2e -run TestFirstRunLoginWithMockGitHubOAuth -count=1 -v

Run the destructive Incus command tier

The incus tier includes tenant create/purge and listing coverage, tenant DNS, the current create and connect CLI machine flows, machine lifecycle, image sync, host override, local trust, and infrastructure checks.

Disposable tenant names are generated from the e2e run id; each tenant's private DNS suffix is that tenant name.

SANDCASTLE_E2E=1 \
SANDCASTLE_E2E_BASE_IMAGE_SOURCE=sandcastle/base:debian-13 \
SANDCASTLE_E2E_AI_IMAGE_SOURCE=sandcastle/ai:debian-13 \
scripts/e2e.sh incus

11. Environment variable reference

Scope and precedence

This is the full environment surface read by Sandcastle code, scripts, and mise tasks. sc-adm and other admin paths load exported env vars plus a local repo .env; exported env vars win over .env. Normal sc user commands ignore local .env and read only saved Sandcastle config plus explicitly exported env vars. Internal rows are generated by Sandcastle for sidecars or machines and normally should not be set by users.
