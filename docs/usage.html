1. Environment setup

Config file (recommended)

Store your defaults in ~/.config/sandcastle/config.yml so you don't need to set env vars on every call.

# ~/.config/sandcastle/config.yml
owner: alice
remote: sc-alice

Once set, sc add myproject/codex works without any extra flags or exports.

Environment variables take precedence over the config file if both are set.

Admin environment variables

Set these before running any sandcastle admin command. These override the config file.

# Incus remote name (default: local)
export SANDCASTLE_REMOTE=big

# Storage pool name in Incus
export SANDCASTLE_STORAGE_POOL=default

# Private CIDR pool for project networks (optional; default 10.248.0.0/16)
export SANDCASTLE_PRIVATE_CIDR_POOL=10.248.0.0/16

# Infrastructure Incus project name (for route broker / shared Caddy)
export SANDCASTLE_INFRASTRUCTURE_PROJECT=sc-infra

# Let's Encrypt email for shared Caddy TLS
export SANDCASTLE_LETSENCRYPT_EMAIL=ops@example.com

Incus certificate selection and admin_remote

sandcastle admin commands always use the global Incus config (~/.config/incus/) which holds admin certificates. All other sandcastle commands use the per-remote Sandcastle config directory (~/.config/sandcastle/<remote>/incus/) which holds the restricted user certificate.

The admin remote is the Incus remote name as it appears in ~/.config/incus/config.yml (e.g. big). It may differ from the Sandcastle user remote (e.g. sc-alice). Configure it once with:

sandcastle config set admin_remote big
# or
export SANDCASTLE_ADMIN_REMOTE=big

When admin_remote is set, sc admin uses it instead of remote. When unset, sc admin falls back to remote and the global Incus config must know that remote name.

Enable verbose output

Set VERBOSE=1 to see the Incus config directory and remote being used, plus per-operation detail from executors.

export VERBOSE=1
sandcastle admin project create alice/myproject --domain myproject.example.com
# [verbose] incus config: ~/.config/incus (default)
# [verbose] incus remote: big

2. Create & manage projects admin

List projects

sandcastle admin project list
sandcastle admin project ls          # alias
sandcastle admin project list --json  # JSON output

Create a project

Provisions an Incus project with isolated Incus feature namespaces, private bridge network, storage volumes, Tailscale sidecar, CoreDNS sidecar, and project-local Sandcastle image aliases.

sandcastle admin project create alice/myproject --domain myproject.dev.example.com
Note: The --domain flag is required. It sets the private DNS zone for all sandboxes in the project (e.g. mysandbox.myproject.dev.example.com).
# Inject an SSH public key into every sandbox at creation time
sandcastle admin project create alice/myproject --domain myproject.dev.example.com --ssh-key "ssh-ed25519 AAAA..."
# Preview what will be created without mutating anything
sandcastle admin project create alice/myproject --domain myproject.dev.example.com --dry-run

Set or update project SSH key

Update the SSH public key stored on an existing project. The new key will be injected into all sandboxes created after this point.

sandcastle admin project set-ssh-key alice/myproject "ssh-ed25519 AAAA..."

Project status

sandcastle admin project status alice/myproject

Delete a project (keep volumes)

Stops runtime resources (sidecars, network) but preserves home/workspace volumes.

sandcastle admin project delete alice/myproject --yes

Purge a project (destroy everything)

sandcastle admin project delete alice/myproject --yes --purge

3. Create users & credentials admin

Create a restricted user certificate

Issues a client certificate for alice and registers it as an Incus restricted trust entry. Prints a bootstrap token.

sandcastle admin user create alice

Example output:

User: alice
Certificate: sc-alice
Remote: sc-alice
Token: 7f3a...
Bootstrap:
  sc remote add sc-alice <join-token> --owner alice
Share the bootstrap commands with the user. They run sc remote add on their machine — this registers the remote and saves it as the default in ~/.config/sandcastle/config.yml.

Grant a user access to a project

Adds the Incus project to the user's certificate restriction list.

sandcastle admin user grant alice alice/myproject
sandcastle admin user grant alice alice/myproject alice/otherproject  # multiple

Issue a new add-token for an existing user

Use this if the user lost their token or you need to re-register the remote.

sandcastle admin user token alice

User bootstrap (on the user's machine)

Run these commands after receiving the bootstrap output from admin.

# One command sets up the remote and writes owner + remote to config
sc remote add sc-alice <join-token-from-admin> --owner alice

# Verify — no env vars needed
sc ls
One-time setup. sc remote add creates ~/.config/sandcastle/sc-alice/incus/ with Incus TLS certs and saves owner + remote to ~/.config/sandcastle/config.yml. All subsequent sc commands pick up the config automatically.

Use incus directly against your remote

sc incus wraps the incus CLI with the correct INCUS_CONF for the active remote.

sc incus ls                          # list instances in your project
sc incus project list               # list Incus projects you can access
sc incus exec sc-alice-myproject -- bash

4. Create & manage containers

Create and enter a sandbox user

Creates an Incus container from the configured base image and drops you into a shell.

sandcastle add alice/myproject/mysandbox

Create in background user

sandcastle add alice/myproject/mysandbox --detach

Choose a template user

Use ai for an AI-tool-equipped image, base for the minimal image.

sandcastle add alice/myproject/ai-box --template ai

Enable nested container tools (Docker/Incus inside sandbox) user

sandcastle add alice/myproject/builder --container-tools

Set a shared home subdirectory user

Multiple sandboxes can share the same home-volume subdirectory (requires --share-home when a conflicting sandbox already exists).

sandcastle add alice/myproject/box1 --home-dir shared
sandcastle add alice/myproject/box2 --home-dir shared --share-home

Expose an app port user

Configures Caddy inside the sandbox to reverse-proxy port 3000 at the sandbox's private URL.

sandcastle add alice/myproject/webapp --app-port 3000

Change app port on an existing sandbox user

sandcastle port set alice/myproject/webapp 8080

Preview the creation plan without creating user

sandcastle add alice/myproject/mysandbox --dry-run

Enter a running sandbox user

sandcastle enter alice/myproject/mysandbox

# Run a specific command
sandcastle enter alice/myproject/mysandbox -- bash -c "echo hello"

Lifecycle commands user

sandcastle start   alice/myproject/mysandbox
sandcastle stop    alice/myproject/mysandbox
sandcastle restart alice/myproject/mysandbox
sandcastle rm      alice/myproject/mysandbox --yes

Inspect a sandbox user

sandcastle inspect alice/myproject/mysandbox
sandcastle inspect alice/myproject/mysandbox --json

List all sandboxes user

sandcastle ls
sandcastle list      # alias
sandcastle ls --json

5. Install local CA trust

Trust the project CA on your machine admin

Makes the project's self-signed CA trusted by your OS/browser so sandbox HTTPS URLs work without warnings.

sandcastle trust install alice/myproject

Remove the CA from local trust admin

sandcastle trust uninstall alice/myproject

Preview trust changes

sandcastle trust install   alice/myproject --dry-run
sandcastle trust uninstall alice/myproject --dry-run

6. Attach Tailscale admin

How it works

Every project gets an sc-tailscale sidecar container created automatically during project create. It sits on the project's private bridge at .2 (e.g. 10.248.0.2) and has a /dev/net/tun device.

The container is started but Tailscale is not activated yet — tailscaled and tailscale up are only run when you explicitly call tailscale up.

Once activated, the sidecar acts as a subnet router: it advertises the entire project private CIDR (e.g. 10.248.0.0/24) to your Tailnet. Any device on your Tailnet can then reach all sandboxes in the project directly by their private IP, without needing Tailscale installed in each sandbox.

Tailscale admin console step: After running tailscale up, approve the advertised route in your Tailscale admin console under the machine's routing settings.

Activate Tailscale on a project (interactive)

Starts tailscaled inside the sidecar and runs tailscale up --advertise-routes=<project-cidr>. Opens an interactive browser auth flow if no auth key is provided.

sandcastle tailscale up alice/myproject

Activate with an auth key (unattended)

sandcastle tailscale up alice/myproject --auth-key tskey-auth-...

Generate auth keys in the Tailscale admin console. Use a reusable key if you need to re-run this after a container restart.

Check Tailscale status

sandcastle tailscale status alice/myproject

Reports the Tailscale state, Tailnet name, assigned IPs, and which routes are being advertised.

Detach Tailscale

Runs tailscale down inside the sidecar. The container keeps running; only the Tailnet attachment is removed.

sandcastle tailscale down alice/myproject

7. Publish public routes

Add a public HTTP route user

Routes external HTTPS traffic for myapp.example.com to the sandbox's app port through the shared Caddy infrastructure.

# Requires SANDCASTLE_ROUTE_BROKER_URL to be set
sandcastle route add myapp.example.com alice/myproject/webapp
DNS proof required: The route broker will tell you which DNS record to create before the route is activated.

List active routes user

sandcastle route list

Remove a route user

sandcastle route rm myapp.example.com

8. Shared infrastructure admin

Create shared infrastructure

Provisions the shared Caddy reverse proxy and route broker inside a dedicated Incus project.

sandcastle admin infra create
sandcastle admin infra create --dry-run   # preview only

Delete shared infrastructure

sandcastle admin infra delete --yes

Serve the route broker

sandcastle admin route-broker serve \
  --listen :9443 \
  --cert   /etc/caddy/certs/broker.crt \
  --key    /etc/caddy/certs/broker.key

9. Manage images admin

Build and upload the AI image (latest tool versions) — recommended

Resolves the latest npm versions of Codex, Claude Code, and Gemini CLI at runtime, builds the OCI image, and imports it into Incus via zstd-compressed tarball.

export SANDCASTLE_REMOTE=big
mise run image:ai:build-upload

Pin specific versions by setting env vars before running:

SANDCASTLE_CODEX_VERSION=0.132.0 SANDCASTLE_CLAUDE_VERSION=2.1.145 SANDCASTLE_GEMINI_VERSION=0.42.0 \
  mise run image:ai:build-upload

Build and upload the base image

export SANDCASTLE_REMOTE=big
mise run image:base:build-upload

All image mise tasks

TaskDescription
mise run image:ai:build-uploadBuild AI image with latest tool versions and upload to Incus
mise run image:base:build-uploadBuild base image and upload to Incus
mise run image:base:buildBuild base image locally only
mise run image:base:uploadUpload a previously built base image
mise run image:depsCheck that docker, incus, tar, zstd are installed

Build an AI image manually (explicit versions)

sandcastle admin image build ai \
  --codex-version  0.132.0 \
  --claude-version 2.1.145 \
  --gemini-version 0.42.0

Build the base image manually

sandcastle admin image build base

Import an OCI image into Incus

sandcastle admin image import base docker://ghcr.io/example/sandcastle-base:latest
sandcastle admin image import ai   docker://ghcr.io/example/sandcastle-ai:latest

Sync an already-imported image to the alias

sandcastle admin image sync sandcastle/base:debian-13
