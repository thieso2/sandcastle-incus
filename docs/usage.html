1. Environment setup

Build binaries

Build native binaries for local use, or cross-compile for handoff to other hosts. Each cross target writes bin/<os>-<arch>/{sandcastle,sc,sandcastle-admin,sc-adm}.

mise run build
mise run build:linux-amd64    # Linux x86-64
mise run build:linux-arm64    # Linux arm64
mise run build:darwin-arm64   # macOS Apple Silicon
mise run build:all           # all three of the above

Install the user CLI

Install only the user-facing sandcastle binary and sc alias. The task installs into ~/.local/bin when that directory is already on PATH; otherwise it installs into /usr/local/bin.

mise run install
sc version

Normal tenant users do not need sandcastle-admin or sc-adm; those are operator tools for tenant, infrastructure, image, and sidecar administration.

Keep the CLI and your sidecar updated (sc update) user

The CLI checks GitHub for a newer release at most once per 24h in the background of normal commands and prints a two-line notice on stderr when one is available (suppressed for dev builds, non-TTY runs, releases younger than 24h, and when SANDCASTLE_NO_UPDATE_NOTIFIER is set). Updates are always user-initiated — never automatic.

sc update --check            # status table only: CLI vs GitHub latest, sidecar vs deployment
sc update                    # show the table, then prompt to apply what's outdated
sc update --yes              # apply without prompting
sc update --version v0.3.1   # pin the CLI to a tag (an older tag rolls back)

The CLI target downloads sandcastle-<os>-<arch>.tar.gz from the release, verifies its SHA-256 against checksums.txt, and atomically replaces the binary, keeping the previous one as .bak. Homebrew-managed installs are never self-replaced — the command prints brew upgrade sandcastle instead. The sidecar target is delegated to the deployment (auth-app token plane, broker mTLS fallback), which pushes its own running binary into your sidecar and restarts only the TLS leaf signer (sub-second; DNS and connectivity untouched).

Config file (recommended)

Store your defaults in ~/.config/sandcastle/config.yml so you don't need to set env vars on every call.

# ~/.config/sandcastle/config.yml
tenant: acme
project: website
remote: sc-alice

Once set, sc create codex works without any extra flags or exports.

Environment variables take precedence over the config file if both are set.

sc config set project website
sc config unset project
sc config show

sc login now records the tenant's project into project: for you, so you rarely set it by hand — see Initial & current project.

sc info prints the active context (tenant, project, remote, auth hostname) and the projects available in the current tenant, marking the current one with *. It fetches the project list live but degrades to the local config (with a note) when the tenant can't be reached, and it never fails on a connectivity error. Use it to discover valid project names for [[project:]machine] references.

sc info
# Tenant:   acme
# Project:  website
# Remote:   sc-alice
# Auth:     auth.example.com
#
# Projects in acme:
#   * website
#     backend

Admin environment variables

Set these before running any sandcastle-admin command, or put them in a local .env file in the current directory. Exported environment variables override .env, and both override the config file for admin commands. For sc-adm install, CLI flags override environment variables, environment variables override the infrastructure seed file, and the seed overrides built-in defaults. Normal sandcastle (sc) user commands ignore local .env files and read only the saved Sandcastle config plus explicitly exported environment variables.

# Incus remote name (default: local)
export SANDCASTLE_REMOTE=big

# Storage pool name in Incus
export SANDCASTLE_STORAGE_POOL=default

# Private CIDR pool for tenant networks (optional; default 10.248.0.0/16)
export SANDCASTLE_CIDR_POOL=10.248.0.0/16

# Prefix for managed tenant Incus projects (optional; default sc)
export SANDCASTLE_INCUS_PROJECT_PREFIX=sc

# Infrastructure Incus project name (shared Caddy / appliances)
export SANDCASTLE_INFRA_PROJECT=sc-infra

# Optional public Auth App hostname reserved by shared Caddy
export SANDCASTLE_AUTH_HOSTNAME=auth.example.com

# GitHub OAuth app credentials for Auth App login
export SANDCASTLE_AUTH_GITHUB_CLIENT_ID=...
export SANDCASTLE_AUTH_GITHUB_CLIENT_SECRET=...

# Initial Sandcastle Admin GitHub usernames, comma-separated
export SANDCASTLE_AUTH_ADMIN_GITHUB_USERS=octocat,hubot

# Optional lab-only device approval bypass user
export SANDCASTLE_AUTH_DEBUG_DEVICE_USER=octocat

# Optional lab/unattended Tailscale sidecar attachment key returned by Device Login
export SANDCASTLE_AUTH_TAILSCALE_AUTHKEY=tskey-auth-...

# Host Incus socket mounted for Auth App provisioning and the broker sidecar (default shown)
export SANDCASTLE_ROUTE_BROKER_INCUS_SOCKET=/var/lib/incus/unix.socket

# Optional prebuilt Linux admin binary uploaded into infrastructure sidecars
export SANDCASTLE_ADMIN_BIN=$PWD/bin/linux-amd64/sc-adm

# Let's Encrypt email for shared Caddy TLS
export SANDCASTLE_LETSENCRYPT_EMAIL=ops@example.com

# Optional exported Caddy ACME data archive restored during install
export SANDCASTLE_INFRA_CADDY_DATA_ARCHIVE=~/.config/sandcastle/infra-caddy-data/big/sc-infra/caddy-data.tgz

# Debug TLS mode: use Caddy's internal CA instead of public ACME
export SANDCASTLE_INFRA_TLS_MODE=internal

Incus certificate selection and admin_remote

sandcastle-admin (sc-adm) always uses the global Incus config (~/.config/incus/) which holds admin certificates. sandcastle (sc) user commands use the per-remote Sandcastle config directory (~/.config/sandcastle/<remote>/incus/) which holds the restricted user certificate. Local repo .env defaults are intentionally admin-only so they cannot redirect sc to the admin remote.

The admin remote is the Incus remote name as it appears in ~/.config/incus/config.yml (e.g. big). It may differ from the Sandcastle user remote (e.g. sc-alice). Configure it once with:

sc-adm config set admin_remote big
# or
export SANDCASTLE_ADMIN_REMOTE=big

When admin_remote is set, sc-adm uses it instead of remote. When unset, sc-adm auto-detects the admin remote by comparing server TLS certificates between the global and per-remote Incus configs.

Enable verbose output

Set VERBOSE=1 to see the Incus config directory and remote being used, per-operation detail from executors, and every Incus REST call with its method, path, status, and duration.

export VERBOSE=1
sc delete tubu --yes
# [verbose] incus config: ~/.config/sandcastle/incus
# [verbose] incus remote: sc-obelix-thieso2-dev
# [verbose] incus api: connect remote sc-obelix-thieso2-dev done (245ms)
# [verbose] incus api: GET /1.0/projects?project=tc2-thieso2-default&recursion=1 -> 200 OK (380ms)
# [verbose] machine delete: tc2-thieso2-default/tubu
# [verbose] machine tubu is Running
# [verbose] incus op: force-stop tubu started
# [verbose] incus api: PUT /1.0/instances/tubu/state?project=tc2-thieso2-default -> 202 Accepted (180ms)
# [verbose] incus op: force-stop tubu done (1.345s)
# [verbose] incus op: delete tubu started
# [verbose] incus api: DELETE /1.0/instances/tubu?project=tc2-thieso2-default -> 202 Accepted (188ms)

Tracing covers every REST call the CLI makes. Two exceptions: the GET /1.0 handshake, which happens while connecting and is reported by the connect remote … done line instead, and websocket traffic (instance exec streams and the event stream behind operation waits) — the request that creates an operation is traced.

2. Create & manage tenants admin

List tenants

sandcastle-admin tenant list
sandcastle-admin tenant list --json  # JSON output

# Show all resources (metadata + machines across all projects) for a specific tenant
sandcastle-admin tenant list acme

Create a tenant

Provisions a tenant Incus project with private bridge network, storage volumes, Tailscale sidecar, CoreDNS sidecar, and tenant-local Sandcastle image aliases.

sandcastle-admin tenant create acme
Note: Machine hostnames are <machine>.<project>.<suffix> where the Tenant DNS Suffix defaults to the tenant name (e.g. codex.default.acme) and can be chosen at creation with --dns-suffix (single label, immutable; also available on sc login for first-login provisioning). Machines in the default project additionally answer at the short alias <machine>.<suffix>; machines in other projects have no short name (ADR-0018). Under ADR-0021 the suffix is also the incus remote name — <suffix> alone, one remote per install (not per project); the project is an orthogonal pin that sc project switch moves. The suffix is unique per install — a first login reserves it, and a second tenant choosing a taken suffix on the same install is rejected.
# Preview what will be created without mutating anything
sandcastle-admin tenant create acme --dry-run

Create a v2 tenant admin v2 MVP

The v2 MVP (see docs/adr/0016-v2-mvp-admin-provisioned-native-incus.md) provisions a tenant for native incus access: one per-tenant infra project sc2-<tenant> with a single sidecar (CoreDNS + Tailscale subnet-router), a shared bridge, and a seeded default app project sc2-<tenant>-default whose profile carries the shared bridge NIC + cloud-init login. Machines resolve flat at <machine>.<tenant>. Prints a restricted Incus certificate-add token the tenant redeems with their own client.

sandcastle-admin tenant create acme --tailscale-authkey tskey-… --ssh-key "ssh-ed25519 AAAA…"
# --cidr-pool defaults to 10.249.0.0/16; --sidecar-image
# takes a system-container base image (alias/fingerprint); --dry-run to preview
No auth key? Omit --tailscale-authkey and the sidecar starts tailscale up interactively; the command prints a https://login.tailscale.com/a/… URL to open and approve, registering the sidecar into your own tailnet. Re-running create after registration is a no-op for Tailscale.
Native use: after incus remote add … --token=…, the tenant runs plain incus launch images:debian/13/cloud web into sc2-acme-default and reaches it at web.default.acme (short alias web.acme — default project only). Cloud-init images are required for login. Add more projects with sandcastle-admin project create acme backend (admin) or, tenant self-service, sc project create backend (after sc login no flags are needed: the request rides the Auth App's token-gated POST /api/projects over the public hostname — tunnel-friendly, no broker port, no client certificate; pass --broker to use the client-cert broker path instead). Under ADR-0021 sc project create does not add a separate per-project remote by default — your install's single <suffix> remote plus sc project switch <name> reaches every project (the switch re-pins the remote). Pass --write-remote to also drop an extra directly-addressable <tenant>-<project>: remote. End-to-end coverage: scripts/e2e-v2.sh.

Set or update tenant SSH key

The user publishes her own SSH public key at login time with sc login --ssh-public-key <path>; the key is baked into the tenant's default-project profile and injected into machines created afterwards. An operator can set or replace the key for a tenant with sandcastle-admin tenant set-ssh-key, or bake one in at creation with tenant create --ssh-key.

sandcastle-admin tenant set-ssh-key acme "ssh-ed25519 AAAA..."

SSH host keys — automatic, authoritative, self-healing

You do not manage host keys. On every sandcastle connect (v2), the CLI reads the machine's SSH host keys over the Incus API — never from the network — and records them in ~/.ssh/known_hosts under the Machine Private Hostname, then connects with StrictHostKeyChecking=yes. Rebuilding a machine changes its host key; the entry is reclaimed automatically, so you never see REMOTE HOST IDENTIFICATION HAS CHANGED. A plain ssh <machine>.<project>.<suffix> uses the same entries and just works.

# every line Sandcastle writes is marked, so it can be reclaimed and purged safely
tubu.default.acme,tubu.acme ssh-ed25519 AAAA... # sandcastle:<remote>/<tenant>

sandcastle ssh-key purge --dry-run   # report what is stale, change nothing
sandcastle ssh-key purge             # drop entries for deleted machines + recycled private IPs
sandcastle ssh-key purge --all --yes # every v2 tenant, no confirmation
What purge will and will not delete. It removes only lines Sandcastle wrote (marked # sandcastle:<remote>/<tenant>) plus untagged literal IP addresses inside the tenant's own private CIDR — recycled DHCP leases with no lasting owner. Your own entries, @cert-authority/@revoked lines, and other hosts are never touched; a wildcard of yours that would shadow a machine name is reported, not removed. The first destructive change each day snapshots the file to ~/.ssh/known_hosts.sc-backup-<date>, and every removal is printed.
Virtual machines. Reading a VM's host key needs incus-agent running inside the guest. When it is not, connect falls back to ssh-keyscan, warns, and marks the entry tofu (trust on first use). Any later connect that can read the machine rewrites the entry and drops the marker. See docs/adr/0020-authoritative-ssh-host-keys.md.

Tenant status

sandcastle-admin tenant status acme

Delete a tenant (keep volumes)

Stops the tenant's runtime resources (sidecar, network) but preserves the durable home/workspace volumes and the Incus projects. Re-run with --purge to destroy everything.

sandcastle-admin tenant delete acme --yes

Purge a tenant (destroy everything)

Removes every app project (machines, images, shared home/workspace volumes, profiles), the infra project (sidecar), and the tenant bridge — scoped to the installation prefix (SANDCASTLE_INCUS_PROJECT_PREFIX), so a same-named tenant of another install on the same daemon is untouched. The sidecar's tailnet device is the tenant's own (BYO tailnet) — remove it in the Tailscale admin console, or use ephemeral keys.

sandcastle-admin tenant delete acme --yes --purge

3. Create users & credentials admin

Create a restricted user certificate

Issues a restricted certificate add-token for alice. Use --tenant to embed tenant access in the token so the user can onboard asynchronously.

sandcastle-admin user create alice --tenant acme

Example output:

User: alice
Certificate: sc-alice
Remote: sc-alice
Token: 7f3a...
Bootstrap:
  sc remote add sc-alice <join-token> --tenant acme
Share the bootstrap command with the user. They run sc remote add on their machine — this registers the remote, saves it as the default in ~/.config/sandcastle/config.yml, and applies the tenant grant embedded in the token.

Grant a user access to a tenant

Tenant access is managed with the tenant commands. Grants add the tenant's Incus project to the user's certificate restriction list.

sandcastle-admin tenant grant acme alice
sandcastle-admin tenant revoke acme alice
sandcastle-admin tenant users acme

Issue a new add-token for an existing user

Use this if the user lost their token or you need to re-register the remote. Include --tenant when the replacement token should be pre-granted.

sandcastle-admin user token alice --tenant acme

Delete a restricted user certificate

Removes the user's restricted Incus certificate entirely. This revokes all tenant access for that certificate. Prompts for confirmation when run interactively; use --yes for scripts.

sandcastle-admin user delete alice --yes

CLI Device Login

Starts browser-assisted sign-in against the Auth App. The CLI prints a verification URL and user code, then polls until the browser approval completes, expires, or is denied.

sandcastle login https://auth.example.com
sandcastle login --dns-suffix acme https://auth.example.com   # first login: choose the Tenant DNS Suffix
sandcastle login --default-project web https://auth.example.com   # first login: name your initial project (replaces "default")
Tenant DNS Suffix & remote name (ADR-0020/0021): at first login the tenant's DNS Suffix (default = tenant name) is reserved on the install and becomes the enrolled incus remote's name — <suffix> alone, one remote per install (ADR-0021), pinned to your initial project; it is immutable, so re-login returns the stored suffix and needs no flag. You can set the suffix two ways: type it into the DNS suffix field on the browser device-approval page, or pass --dns-suffix on the CLI — when both are given, the CLI flag wins. A suffix already claimed by another tenant on that install is rejected. On re-login, the tenant's legacy / per-project remotes for this install are lazily collapsed to the single <suffix> remote (endpoint-scoped; the current remote is never removed).
Initial project name (issue #93): at first login you may also name your tenant's one project instead of the default default. It becomes the middle part of machine hostnames (<machine>.<project>.<suffix>), the project the install's single <suffix> remote is initially pinned to, and the holder of the DNS short-alias. Set it two ways — the Initial project field on the browser device-approval page or --default-project (alias --initial-project) on the CLI, CLI wins. Unlike the suffix it is not immutable; blank ⇒ default, and re-login reuses the stored name.
Current project stored on login: after a successful login sc writes the tenant's project into ~/.config/sandcastle/config.yml (project:) so bare machine references — sc c <machine>, sc create <machine> — resolve without --project or a manual sc config set project. If the tenant has a single project it is stored silently; if it has several, an already-valid configured choice is kept, otherwise you are prompted to pick one (a non-interactive login defaults to the tenant's current project and prints a note). Passing --default-project selects it directly without a prompt. Inspect the result with sc info.
Approval page adapts to your tenant: the browser device-approval page only shows the DNS suffix and Initial project inputs on your first login, when there is nothing to name yet. Once your tenant exists, the page instead displays your existing (immutable) DNS suffix and current projects read-only — there is nothing to fill in, so re-approving is a single click. The CLI flags (--dns-suffix, --default-project) behave the same regardless.

Idempotent (no browser when already authorized). Before starting a device flow, login checks whether you already hold a working login for the requested Auth Hostname — the saved CLI Auth Token is still accepted by the Auth App, the enrolled Incus remote responds, and the tenant subnet is reachable over the tailnet. If so it reports Already logged in and exits, no web. Crucially this is resolved from the per-install credential maps (installs / auth_tokens in your config), not just the currently-active install: running sc login <other-host> when you already have valid credentials for it skips the browser and switches your active install to it (re-pointing auth hostname, token, broker, remote, and the shared incus current remote). Pass --force to re-authenticate anyway.

Tailnet precheck. Unless --skip-setup is given, login refuses to start the device flow on a machine that is not itself a Tailscale node (tailscale missing, logged out, or stopped): the v2 remote lives at the sidecar's tailnet IP and tenant machines sit behind its subnet route, so completing login would leave the tenant unreachable. It also refuses when no incus client is on PATH (enrollment shells out to incus remote add). After enrollment the routing verification prints a per-layer diagnosis (tailscale up → accept-routes → route offered/primary → probe egresses via the tailnet) and halts with advice specific to the first broken layer.

Host preparation. sudo sc-adm install-incus puts the latest Incus (Zabbly stable) on a Debian-based host: apt repository, install/upgrade, incus admin init --minimal when uninitialized, and adds the invoking user to incus-admin. Idempotent; --codename overrides the detected apt suite, --skip-init leaves the daemon untouched. A fresh box is two commands: sudo sc-adm install-incus, then sc-adm install ….

One-command server install. sc-adm install deploys the complete server side — the Auth App appliance and the broker appliance — in one command with a shared tenant CIDR pool (--cidr-pool, optional). --prefix (default sc) scopes the installation's project and appliance names, so several sandcastles can share one Incus host; the command refuses to run when an installation under the same prefix already exists and warns when the pool overlaps a host address. The separate auth-app deploy and bootstrap commands remain for piecewise installs.

Integrated public ingress (--ingress). The auth-app appliance can terminate its own public hostname — caddy (and cloudflared) run inside the auth-app container, proxying to the service on localhost, so there is no separate edge appliance and no cross-container IP wiring. Modes: acme — caddy binds the host's :80/:443 (must be free; preflight-checked) and gets a real Let's Encrypt certificate; you provide an A record for the hostname and optionally --acme-email. cloudflare — cloudflared dials out to a Cloudflare tunnel, no inbound ports at all; pass either --cloudflare-tunnel-token (dashboard-created tunnel: create a Cloudflared tunnel, copy the connector token, add a public hostname routing to http://localhost:8080 — the DNS record is created automatically) or --cloudflare-api-token (Tunnel:Edit + DNS:Edit + Zone:Read; the installer creates the tunnel, its ingress rule, and the proxied DNS record itself). none (default) — bring your own edge, e.g. sc-edge.

sc-adm install --auth-hostname burg.example.dev --github-client-id … --github-client-secret … --admin-github-users you

BYO tenant tailnet. The tenant sidecar joins your tailnet. The normal flow is interactive: log in with no key — login prints the sidecar's login.tailscale.com URL and waits; open it, approve the advertised subnet route, and login continues automatically (the sidecar's Incus Reach completes on the next poll). The URL survives re-polls and sc login re-runs, and a fresh one is printed if it changes. --tailscale-auth-key tskey-… is for unattended (CI) logins only. The auth-app's --tailscale-auth-key is only an optional deployment-wide default for tenants that bring no key.

Login prepares the User SSH Public Key used for Machine SSH Access before starting the device flow. Pass --ssh-public-key <path> to choose a public key explicitly. Without that flag, the CLI uses ~/.ssh/id_ed25519.pub when it exists, otherwise it creates a local Sandcastle SSH Key at ~/.ssh/sandcastle_ed25519. The private key stays local. On approval, the Auth App stores the public key, reconciles it onto existing Personal Tenant machines, and writes it to Personal Tenant metadata so future machines receive the key during bootstrap.

sandcastle login --ssh-public-key ~/.ssh/id_ed25519.pub https://auth.example.com

After browser approval, the Auth App creates or ensures the user's Personal Tenant, ensures its Default Project, returns a restricted Incus certificate add token plus a CLI Auth Token, and streams provisioning status through CLI polling. The CLI enrolls the remote with its own local client key, remembers the Auth Hostname and CLI Auth Token for later Auth App API calls, and sets the default tenant only when exactly one accessible Tenant is returned.

For a single accessible tenant, CLI login enrolls the restricted Incus remote, applies tenant DNS, installs local resolver config, installs the tenant CA into local trust, and runs tailscale up in the tenant Tailscale sidecar. These post-login steps use the newly enrolled Incus config.

When the CLI Login Result includes a Tenant Tailnet, login verifies the local Tailscale client is connected to that selected Tenant Tailnet and has a Tailscale IP before reporting Login Readiness. Multi-tenant login does not silently join every accessible Tenant Tailnet.

Set VERBOSE=1 to print login step timings for SSH key preparation, device start and poll requests, Incus remote enrollment, DNS, trust, Tailscale, and local tailnet verification.

Login Readiness means the Incus credential, User SSH Public Key, Personal Tenant, Default Project, Tenant Tailnet, local DNS, local tenant CA trust, local Tailscale Prerequisite, and local Sandcastle config are ready. A successful first-run login prints the next command, currently sandcastle create dev.

User bootstrap (on the user's machine)

Run these commands after receiving the bootstrap output from admin.

# One command sets up the remote and writes tenant + remote to config
sc remote add sc-acme <join-token-from-admin> --tenant acme

# Verify — no env vars needed
sc list
One-time setup. sc remote add creates ~/.config/sandcastle/sc-acme/incus/ with Incus TLS certs and saves tenant + remote to ~/.config/sandcastle/config.yml. All subsequent sc commands pick up the config automatically.

Select an accessible tenant user

Users with Tenant Access to more than one tenant can list accessible tenants and select the local Current Tenant without editing config directly. Switching validates Tenant Access through the Auth App by default and changes only Sandcastle local config; it does not change the Current Project, mutate Incus CLI state, install DNS/trust, or join a Tenant Tailnet. Use --local-only only for bootstrap or offline debugging when Auth App validation is intentionally unavailable.

sc tenant list
sc tenant switch skorfman
sc tenant switch skorfman --local-only

Use incus directly against your remote

Two sc incus* variants target the two Incus projects that make up a tenant. Each wraps the incus CLI with the correct INCUS_CONF and INCUS_PROJECT. Pass an explicit --project to override the project. sc incus requires a live tenant and reads the app project name off the tenant summary.

Switching between sandcastles: use sc remote list (alias ls) to see the enrolled installs — one <suffix> remote per install under ADR-0021, with its project pin and Auth Hostname, the active one marked * — and sc remote switch <name> (alias use) to move the whole sc CLI (remote, tenant scoping, and the auth plane: Auth Hostname, broker, and CLI token are all re-pointed to that install). It writes both ~/.config/sandcastle/config.yml and the shared incus dir's current remote, so sc, sc incus, and raw incus <remote>: agree. sc config set remote … is the low-level equivalent; sc incus remote switch only moves the raw incus passthrough; SANDCASTLE_REMOTE overrides per invocation. The switch also re-pins the project to the target install's own (derived from that remote's incus project pin) and reports it — Switched to remote "obelix" (project "work") — so sc ls/sc c work immediately; use sc project switch <name> to pick a different project within the install.

CommandProjectContents
sc incus<prefix>-{tenant}-{project}All tenant machines (created with sc create or plain incus launch)
sc incus-infra<prefix>-{tenant}The tenant sidecar (Tailscale, CoreDNS); most restricted certificates cannot see it
 
sc incus ls                          # list machines in the app project
sc incus project list               # list Incus projects you can access
sc incus exec web -- bash
sc incus launch images:debian/13/cloud web  # launch straight into the app project
sc incus-infra ls                   # inspect the sidecar

Instances in the app project inherit its default profile, which attaches the shared bridge NIC, the login user + SSH key, and the shared /home and /workspace volumes — so a bare incus launch images:debian/13/cloud web (cloud image) comes up networked and login-ready, reachable at web.<suffix>. The sidecar CoreDNS zone is the DNS authority for the suffix (ADR-0018): the auth-app registers every machine's record within seconds of an instance lifecycle event, with a 30s reconcile pass as safety net.

4. Create & manage machines

Manage lightweight projects user

Projects are namespaces inside the current tenant. They do not create separate Incus projects, and restricted users can manage them after the tenant token is accepted.

sandcastle project list                    # (alias ls) marks the current project with *
sandcastle project switch website          # select the local current project (like `incus remote switch`)
sandcastle project create website
sandcastle project status website
sandcastle project set-cloud-identity default gcp
sandcastle project unset-cloud-identity default
sandcastle project set-docker-autostart website on
sandcastle project delete website --yes

project switch <name> is the durable way to change the active project (it writes project: to ~/.config/sandcastle/config.yml, mirroring incus remote switch); it validates the project exists in the current tenant, or pass --local-only to set it without the lookup. Under ADR-0021 it also re-pins the active install's single incus remote to the new project, so sc, sc incus, and raw incus <remote>: all agree — no per-project remotes, no divergence. project list (alias ls) marks the current one with *. Per-command overrides remain the reference prefix (sc c <project>:<machine>) and SANDCASTLE_PROJECT; precedence is env → config → default.

A project's default Cloud Identity Config is used by sandcastle create and by sandcastle connect when it auto-creates a missing machine. sandcastle project set-cloud-identity validates that the named Auth App Cloud Identity Config exists for the current tenant before it updates project metadata; a same-named config from another tenant is refused. The Docker autostart setting controls whether Docker is enabled and started during new machine configuration; it also enables nested container tooling for those new machines. Docker is installed in the base image but the daemon is disabled by default. The default project can carry both settings too. project delete prompts for confirmation when run interactively; use --yes for scripts and other non-interactive runs.

Create and connect to a machine user

Creates an Incus container machine from a stock cloud image. create prints the machine's IP and SSH hint but does not auto-connect; connect (alias c) creates the machine when missing and opens a shell.

Machine references accept machine, project:machine, tenant/machine, and tenant/project:machine. Use the tenant-qualified forms to work in another accessible tenant without changing the Current Tenant.

Machines use private tenant-network IPs. Tailscale is attached at the tenant sidecar, which advertises the tenant CIDR; containers do not need their own Tailscale IPs. Machine configuration stops and masks tailscaled even though the base image includes the Tailscale client package for sidecar reuse.

sandcastle create website:codex
sandcastle create skorfman/default:codex

One-command workflow. sc connect (alias c) is the fastest path: it creates the machine when it doesn't exist (default image; pass --vm to create it as a virtual machine — sc c --vm dev), starts it when stopped, waits for sshd (VMs get a longer boot window), and opens an SSH session as the profile login user with the login SSH key — sc c dev is all a new machine needs. The lifecycle commands (delete/start/stop/restart) act on those instances (delete force-stops first). create launches a stock cloud image into the tenant's app project: default image images:debian/13/cloud (override with --image), --vm launches a virtual machine, and the project default profile supplies the login user (your client Unix username from login, falling back to the deployment default, then dev) + SSH key with sshd enabled, the shared bridge NIC, and the shared /home and /workspace volumes (per-project filesystem volumes attached to every CT and VM, so machines in a project share home directories and a workspace). DNS auto-registers as <machine>.<suffix>. create accepts only --image, --vm, and --dry-run, and does not auto-connect — it prints the machine's IP and SSH hint. sc list shows every instance in the tenant's app projects (including machines launched with plain incus launch) with its type (CT or VM), and sc incus scopes to the app project (sc incus-infra to the tenant's infra project, which most restricted certificates cannot see).

sandcastle create dev2 --vm --image images:ubuntu/24.04/cloud

Connect to a machine user

sandcastle connect dev                  # current install + current project
sandcastle connect website:codex        # current install, project "website"
sandcastle c obelix:website:codex       # install "obelix", project "website"
sandcastle c --vm dev            # create it as a VM if it doesn't exist yet

# Run a specific command
sandcastle connect website:codex -- bash -c "echo hello"
Reference grammar (ADR-0020): [[dns-suffix:]project:]machine. The colon count selects scope — 0 = current install + current project, 1 = current install + named project, 2 = named install (by its Tenant DNS Suffix) + project. A single colon is always project:machine (there is no silent incus-native remote:instance fallback). Naming another install switches to its incus remote (<suffix>, one per install; the named project is applied per-call) automatically; if you're not logged into that install, connect errors with sc login guidance rather than auto-provisioning. The same grammar is accepted by create, the machine-lifecycle commands, and image save.

connect starts a stopped machine before refreshing known hosts or handing off to SSH. It uses SSH to the machine's private IP and uses the private machine hostname as the host-key alias. Local DNS is useful for private URLs, but it is not required for Machine SSH Access.

If the machine is missing, connect logs that it is creating the machine, then connects after creation. Pass --vm to create it as a virtual machine.

Before handing off to SSH, connect probes TCP port 22 with a 2-second timeout when the connection plan came from the local cache. If the host is unreachable (e.g. tenant was recreated with a new IP), the cache entry is invalidated and the plan is resolved fresh from Incus, so connect self-heals without manual intervention.

Fix machines user

sandcastle fix   dev
sandcastle fix   dev --check
sandcastle fix   obelix:web:api --only agent-forwarding

fix applies idempotent maintenance fixups to a running machine over SSH (as the login user, via sudo). Cloud-init only runs at a machine's first boot, so a machine built before a fixup shipped never receives it; fix backfills the change in place. It resolves the same [[remote:]project:]machine reference as connect (creating/starting the machine if needed). --check reports status and changes nothing; --only <name> limits it to the named fixup(s).

Registered fixups:

  • agent-forwarding — installs the forwarded-SSH-agent indirection (/etc/ssh/sshrc republishes each session's agent at ~/.ssh/ssh_auth_sock; /etc/zsh/zshrc and /etc/bash.bashrc export it) so ssh -A keeps working inside a herdr/tmux pane whose server outlives the ssh session. New machines get this from cloud-init; fix is the backfill for older ones. It also reports (never rewrites) a broken hand-rolled ~/.zshrc that republishes through the consumed path.
Fleet backfill. To apply a fixup across every machine in a project at once (admin, over incus exec, no SSH/agent needed), use scripts/fix-agent-forwarding.sh --remote <r>: --project <p>.

Publish public routes

Requires route ingress. sc route works only on installs where native-ACME route ingress is enabled — deploy with --route-ingress acme (on sc-adm install or sc-adm auth-app deploy). Caddy then binds the host's :80/:443 and gets real Let's Encrypt certificates for route hostnames. This is independent of the Auth Hostname's own ingress: routes can run beside a Cloudflare-tunnelled login host — e.g. login on --ingress cloudflare (home.thieso2.dev) and routes on --route-ingress acme --route-base-domain home.tc42.uk, served by the same appliance Caddy. By default routes live under the Auth Hostname; --route-base-domain puts them under a separate domain. The operator provisions a wildcard *.<route-base-domain> → the host's public IP (a wildcard CNAME to a dynamic-DNS name works — the record tracks the IP). On an install without route ingress every sc route command returns a clear error telling you to enable it.

Publish a machine's port user

Exposes an HTTP app running inside one of your machines to the public Internet through the Auth App's Caddy. The default public hostname is <name>.<tenant>.<auth-hostname> (the <tenant> segment is fixed to your tenant, so it can't collide with another tenant); --name defaults to the machine name. The certificate is issued by Let's Encrypt on the first request, so the very first hit is slow.

# Publish port 3000 of machine "web" → https://web.<tenant>.<auth-hostname>
sc route publish web --port 3000

# Choose the subdomain label
sc route publish web --port 3000 --name api

# Bring your own domain: publish it, then CNAME it onto the auth host yourself
sc route publish web --port 3000 --hostname app.customer.com

The machine is addressed with the usual [[remote:]project:]machine syntax. --port (the backend port inside the machine) is required. Custom hostnames are self-serve first-come; sc does not automate DNS — a custom route only serves once you point its DNS at the auth host and the certificate is issued.

List, inspect, and delete routes user

sc route list                                  # HOSTNAME · MACHINE · PORT · STATUS
sc route status web.acme.sc2.example.dev       # backend, URL, health
sc route delete web.acme.sc2.example.dev --yes

STATUS is live (machine running, serving), awaiting-dns (custom hostname not yet resolving / certificate pending), or unhealthy (backing machine stopped or unreachable). Routes self-heal: if a machine changes IP on reboot the route is refreshed automatically, and if a machine is deleted its routes are pruned. Add --json to list/status for machine-readable output.

6. Attach Tailscale

How it works

Every tenant gets an sc-tailscale sidecar container created automatically during tenant create. It sits on the tenant's private bridge at .2 (e.g. 10.248.0.2) and has a /dev/net/tun device.

The container is started but Tailscale is not activated yet — tailscaled and tailscale up are only run when you explicitly call tailscale up. The shared base image keeps tailscaled.service masked by default; the sidecar setup unmasks it for this role only.

Once activated, the sidecar acts as a subnet router: it advertises the entire tenant private CIDR (e.g. 10.248.0.0/24) to your Tailnet. Any device on your Tailnet can then reach all machines in the tenant directly by their private IP, without needing Tailscale installed in each machine.

Tailscale admin console step: After running tailscale up, approve the advertised route in your Tailscale admin console under the machine's routing settings.

Activate Tailscale on a tenant (interactive)

Starts tailscaled inside the sidecar and runs tailscale up --timeout=60s --advertise-routes=<tenant-cidr>. Without an auth key, Tailscale may print a browser auth URL; the command is still bounded by the timeout so unattended flows fail cleanly.

sandcastle tailscale up
sandcastle tailscale up acme  # explicit tenant

Activate with an auth key (unattended)

sandcastle tailscale up --auth-key tskey-auth-...

Generate auth keys in the Tailscale admin console. Use a reusable key if you need to re-run this after a container restart.

Check Tailscale status

sandcastle tailscale status

Reports the Tailscale state, Tailnet name, assigned IPs, and which routes are being advertised.

Detach Tailscale

Runs tailscale down inside the sidecar. The container keeps running; only the Tailnet attachment is removed.

sandcastle tailscale down

7. Auth App service admin

How shared infrastructure is deployed

The Auth App and the Sandcastle Broker run as appliances on the Incus host. Deploy them together with sc-adm install (see §3), or piecewise with sc-adm auth-app deploy and sc-adm bootstrap. This section documents how the running Auth App behaves once deployed.

The Auth App runs sandcastle-admin auth-app serve with a persistent SQLite database at /var/lib/sandcastle/auth/auth.db. When an Auth Hostname is configured, the appliance's ingress routes it to the Auth App. CLI Device Login provisions Personal Tenants only after browser approval, not at allowlist add time or browser-only GitHub login time. During CLI Device Login, the user CLI sends the local OS username and the Personal Tenant stores it as the Unix user for future Machines. --username and SANDCASTLE_AUTH_DEFAULT_UNIX_USER are fallback defaults for clients that do not send a local Unix username.

Verbose logging & the Activity Log. The Auth App and the broker write a verbose, timestamped line for every HTTP request and every unit of work (named "spans" such as provision.personal_tenant, project.create, issue workload token), each with its duration, to stderr — visible with journalctl -u sandcastle-auth-app / journalctl -u sandcastle-broker. The Auth App additionally persists its entries to the logs table in auth.db and exposes them at / → Activity log (https://<auth-hostname>/logs). Visibility is scoped by identity: a Sandcastle Admin sees every user's entries (plus system entries with no user), while every other signed-in user sees only her own. The page supports a free-text filter (?q=) over event/path/detail and a row cap (?limit=, default 500). Log rows are retained indefinitely (no automatic pruning). Broker activity currently appears only in the broker's own journald stream, not in the Activity Log.

Initial Sandcastle Admins from SANDCASTLE_AUTH_ADMIN_GITHUB_USERS can sign in through GitHub and manage the Login Allowlist at /admin/allowlist and Tenant Access at /admin/access. The allowlist accepts explicit GitHub users only; password login is not available. For disposable lab integrations, SANDCASTLE_AUTH_DEBUG_DEVICE_USER enables POST /debug/device/approve to approve CLI Device Login as that allowlisted user without a browser session. SANDCASTLE_AUTH_TAILSCALE_AUTHKEY can be set at deployment to return a Tailscale auth key to approved CLI Device Login clients for unattended tenant sidecar attachment; this is intended for disposable integration environments and is not stored in Incus metadata.

The Auth App also serves per-tenant OIDC provider metadata at /t/<tenant>/.well-known/openid-configuration and per-tenant public signing keys at /t/<tenant>/.well-known/jwks.json. Legacy global metadata remains available at /.well-known/openid-configuration and /.well-known/jwks.json. Private signing keys are stored encrypted in the Auth Database. See Sandcastle OIDC Configuration for the full provider and machine token flow.

Users can manage User-owned, tenant-scoped Cloud Identity Configs at /cloud-identities. Use sc cloud-identity gcp setup --tenant <tenant> after sc login to configure tenant-scoped GCP Workload Identity Federation, save the Auth App config named gcp for that tenant, and print the audience and service account impersonation URL for confirmation. Set a project default with sc project set-cloud-identity default gcp; project defaults are accepted only when the named config exists for the current tenant.

Serve the Auth App

sandcastle-admin auth-app serve \
  --listen :9444 \
  --database /var/lib/sandcastle/auth/auth.db \
  --auth-hostname auth.example.com \
  --github-client-id ... \
  --github-client-secret ... \
  --admin-github-users octocat,hubot \
  --debug-device-user octocat \
  --simulate-github-token <shared-secret>

The debug device user flag is intended only for disposable integration environments. It registers POST /debug/device/approve, which approves a pending device code as the configured allowlisted user. Use --tailscale-auth-key only for unattended lab login flows where approved CLI clients should receive a Tailscale auth key for tenant sidecar attachment.

Simulated GitHub mode (dev/e2e only). Passing --simulate-github-token <secret> (also accepted by auth-app deploy; env SANDCASTLE_AUTH_SIMULATE_GITHUB_TOKEN) lets the Auth App fabricate GitHub logins with no real OAuth app — --github-client-id/--github-client-secret become optional. It registers GET/POST /oauth/github/simulate, which (gated by a constant-time comparison against the shared secret) allowlists the given username, mints the browser session, and optionally approves a pending device user_code. Drive it from the CLI with sc login <auth-host> --simulate-token <secret> --as <username>. Never enable in production — anyone holding the token can authenticate as any user. A wrong token returns 403; an unset token leaves the route unregistered (404).

Update the deployment (sc-adm update)

Updates this install's global components — the auth-app and broker appliances — from a GitHub release: it downloads the matching linux-<arch> asset per appliance, verifies it against checksums.txt, pushes it into each appliance, restarts the service, and stamps user.sandcastle.binary-version into the instance config. Idempotent — re-run to repair a partial update. Components are scoped by the installation prefix, so a neighbour install sharing the host is untouched. The image-builder appliance carries no sandcastle binary and is not updated.

sc-adm update --check                 # fleet version table from the stamps (works for stopped instances)
sc-adm update                         # update auth-app + broker to the latest release
sc-adm update --version v0.3.1        # pin (an older tag rolls back)
sc-adm update --tenants alice,bob     # also force-roll these tenants' sidecars
sc-adm update --all-tenants           # force-roll every tenant sidecar (e.g. a security fix)

Tenant sidecars are normally tenant-managed: each tenant runs sc update, which the deployment serves by pushing its own running binary — so sidecars can never run ahead of the deployment. --tenants/--all-tenants is the operator override for security fixes and abandoned tenants. Appliances with no version stamp (deployed before the update system) show unknown and are treated as outdated; their first update self-heals the stamp. The auth-app also shows an always-visible version card on its web pages: green when current, amber with the sc-adm update command and a release-notes link when behind.

Version skew. The CLI sends its version on every auth-app/broker call and responses carry the appliance version: a mismatch prints a one-line warning, never a block. A known-breaking release can set a minimum CLI version, in which case older CLIs get a clean “run sc update” refusal instead of protocol errors.

Manage the Login Allowlist

Open https://auth.example.com/admin/allowlist as a Sandcastle Admin. Adding a user verifies the GitHub username and stores the normalized username plus GitHub numeric account ID. Removing a user blocks future GitHub and CLI login, revokes their restricted Incus certificate grants, and removes their User SSH Public Key from Machines without deleting their Personal Tenant or Machines.

Manage Tenant Access

Open https://auth.example.com/admin/access as a Sandcastle Admin to view Users, Tenants, and restricted Incus certificate grants. Grant and revoke actions use the same project grant model as sandcastle-admin tenant grant and sandcastle-admin tenant revoke.

Image Builder appliance (build in a container, publish to GHCR)

Instead of building locally with docker, the Image Builder builds Sandcastle Images inside a dedicated, admin-managed Incus container and publishes them to GHCR. It is an infrastructure appliance, not a Machine. See docs/adr/0010-image-builder-appliance.md.

export SANDCASTLE_REMOTE=big
export SANDCASTLE_GHCR_TOKEN=<github PAT with write:packages>
mise run image:all:build-remote          # or sc-adm image build-remote base|ai|all

How it works: the sc-build project holds a persistent sc-builder container (neutral images:debian/13, security.nesting, /dev/fuse) that builds with rootless podman against a cache volume. build-remote ensures the appliance and a one-time ghcr OCI remote, ships the working-tree build context in, logs in with the token over stdin (never persisted), builds :latest + an immutable :<git-describe> tag, pushes to GHCR, then copies the immutable tag into the local Incus alias and syncs tenant aliases. The AI image is built FROM the immutable base tag of the same run.

Provisioning is decoupled from building. On a cold start (no appliance) build-remote auto-provisions; on a warm appliance it only starts it and builds, skipping the (idempotent) provision step for speed. Provision deliberately with mise run image:builder:provision (or sandcastle-admin image builder provision) — e.g. after changing the provision toolchain. Builds are heavily cached: the Dockerfiles use RUN --mount=type=cache for apt/npm and the build runs with BUILDAH_LAYERS=true, so a no-op rebuild is seconds. Pass VERBOSE=1 to stream the underlying podman build/push output live.

Flags: --require-clean (refuse a dirty tree), --no-push, --no-import, --rebuild-builder, --ghcr-repo, --ghcr-user, --remote, --platform, --dry-run. Manage the appliance with sandcastle-admin image builder provision, sandcastle-admin image builder status, and sandcastle-admin image builder destroy [--yes] [--keep-cache] (prompts for confirmation when run interactively). GHCR packages must be public-read so the Incus host pulls without credentials; the token is only used to push.
