Architecture Overview

Every Sandcastle project consists of a set of Incus resources provisioned and managed by this codebase. The diagram below shows the full set of objects created for a single project, and the objects created inside a sandbox.

┌─────────────────────────────────────────────────────────────────────┐ │ Incus server (remote: big) │ │ │ │ ┌──────────────────────────────────────────────────────────────┐ │ │ │ Incus project: sc-owner-proj (features: images, profiles, │ │ │ │ storage.buckets, storage.volumes) │ │ │ │ │ │ │ │ Profiles: container { root→pool, eth0→private bridge } │ │ │ │ │ │ │ │ Volumes: sc-home (home directory, persists) │ │ │ │ sc-workspace (workspace, persists) │ │ │ │ sc-ca (project CA, persists) │ │ │ │ │ │ │ │ Sidecars: sc-owner-proj (tailscale, addr .2) │ │ │ │ sc-dns (CoreDNS, addr .53) │ │ │ │ │ │ │ │ Sandbox: sc-boxname (user container) │ │ │ │ ├─ eth0 → private bridge (10.248.x.0/24) │ │ │ │ ├─ home → sc-home volume │ │ │ │ ├─ workspace → sc-workspace volume │ │ │ │ └─ Caddy (443) + sshd (22) inside │ │ │ └──────────────────────────────────────────────────────────────┘ │ │ │ │ ZFS Pool: sc-owner-proj (sub-dataset of admin pool) │ │ Network: sc-c8bd3728 (bridge, global namespace) │ └─────────────────────────────────────────────────────────────────────┘

Project Lifecycle Tests

TestDisposableProjectCreateAndPurge

project_lifecycle_test.go
SANDCASTLE_E2E=1

Creates a complete Sandcastle project — including its per-project ZFS storage pool, private bridge network, sidecars (tailscale + CoreDNS), durable volumes, and the container profile. Validates all resources exist. Runs a second create call to prove idempotence. Then purges everything and verifies the storage pool is gone.

CREATE owner/project ↓ incusx.ProjectCreator ├─ ensureProject → Incus project (features: images, profiles, storage.volumes, storage.buckets) ├─ ensureStoragePool → ZFS pool sc-owner-project (sub-dataset of admin pool) ├─ ensureProjectImages → copy base + AI image into project namespace ├─ ensurePrivateNetwork → bridge sc-xxxxxxxx (10.x.x.0/24) ├─ ensureStorageVolumes → sc-home, sc-workspace, sc-ca ├─ ensureProjectCA → write CA cert/key into sc-ca ├─ ensureSidecars → tailscale + CoreDNS containers ├─ ensureContainerProfile → "container" profile {root→pool, eth0→bridge} └─ ensureDNSFiles → CoreDNS zone files + restart ASSERT ✓ project appears in project.List() ✓ storage pool sc-owner-project exists via GetStoragePool() ✓ "container" profile exists with root.pool = sc-owner-project ✓ second CreateProject() call succeeds (idempotent) PURGE (project.DeletePlan{Purge:true}) ├─ delete all instances (stop + delete) ├─ delete non-default profiles (container) ├─ delete private network ├─ delete durable volumes ├─ delete project images ├─ delete Incus project └─ delete ZFS storage pool ASSERT AFTER PURGE ✓ GetStoragePool(sc-owner-project) → 404
Resources
1 Incus project, 1 ZFS pool, 1 bridge, 3 volumes, 2 sidecar instances, 1 profile, copied images
Duration
~35 s
Cleanup
Full purge in t.Cleanup

Sandbox Lifecycle Tests

TestSandboxLifecycleE2E

sandbox_lifecycle_test.go
SANDCASTLE_E2E=1 BASE_IMAGE + AI_IMAGE

The most comprehensive test. Exercises the complete sandbox lifecycle: create, update port, proxy validation, lifecycle actions (stop/start/restart/remove), durable volume persistence, and sandbox recreation.

SETUP sync sandcastle/base → sandcastle/base:<runID>-sandbox sync sandcastle/ai → sandcastle/ai:<runID>-sandbox create project owner/project (full project with sidecars) SANDBOX CREATE owner/project/boxname ↓ sandbox.PlanCreate → resolve image, pick IP from CIDR SandboxCreator.CreateSandbox ├─ create instance sc-boxname (from AI image) ├─ configure eth0 → private bridge (10.x.x.20) ├─ mount home → /home/<linux-user> ├─ mount workspace → /workspace ├─ write Caddyfile (HTTPS reverse proxy → 127.0.0.1:3000) ├─ write TLS cert + key (signed by project CA) └─ start instance ASSERT after CREATE ✓ instance exists ✓ Caddyfile contains hostname + port 3000 ✓ TLS cert valid for sandbox hostname ✓ private key is valid PEM ✓ Python HTTP server started → Caddy proxies HTTPS → "sandcastle-app-3000" PORT UPDATE 3000 → 5173 sandbox.PlanSetPort → SandboxPortSetter.SetAppPort └─ rewrites Caddyfile + cert ASSERT after PORT UPDATE ✓ Caddyfile now shows port 5173 ✓ HTTPS proxy returns "sandcastle-app-5173" LIFECYCLE ACTIONS (in order) stop → instance StatusCode = Stopped start → instance StatusCode = Running restart → instance StatusCode = Running (new boot) remove → instance deleted ASSERT after REMOVE ✓ GetInstance(sc-boxname) → 404 ✓ sc-home volume still exists (durable) ✓ sc-workspace volume still exists (durable) ✓ sc-ca volume still exists (durable) SANDBOX RECREATE same ref owner/project/boxname ↓ CreateSandbox (reuses existing volumes) ✓ instance exists again ✓ ingress files written correctly
Resources
2 image aliases, 1 project (full), 1 sandbox instance
Duration
~60–70 s
Cleanup
Full purge of project + aliases in t.Cleanup

DNS Tests

TestProjectDNSE2E

project_dns_test.go
SANDCASTLE_E2E=1 BASE_IMAGE + AI_IMAGE

Verifies that CoreDNS inside the project correctly resolves sandbox hostnames, both exact and wildcard, and that the zone is updated when a sandbox is removed.

SETUP project: owner/dns sandbox1: owner/dns/codex → hostname codex.dns-<runID>.e2e.project-tld sandbox2: owner/dns/claude → hostname claude.dns-<runID>.e2e.project-tld ASSERT DNS (via exec inside sc-dns instance) dig @10.x.x.53 codex.dns-<runID>.e2e.project-tld → A 10.x.x.20 dig @10.x.x.53 *.dns-<runID>.e2e.project-tld → wildcard → 10.x.x.20 REMOVE sandbox1 PlanLifecycle(ActionRemove) → instance deleted REFRESH DNS PlanRefreshDNS + RefreshDNS → CoreDNS zone updated ASSERT DNS AFTER REMOVAL dig @10.x.x.53 codex.dns-<runID>.e2e.project-tld → NXDOMAIN
Duration
~60 s
Cleanup
Full purge

TestLocalDNSInstallForwardRefreshUninstallE2E

local_dns_test.go
SANDCASTLE_E2E=1

Tests the host-side local DNS forwarder (no real Incus instances). Uses in-memory UDP responders to simulate project DNS endpoints.

INSTALL project DNS resolver via localdns.Installer → writes state file → starts resolver goroutine (listens on 127.0.0.1:53540) QUERY codex.project.e2e.project-tld → forwarded to UDP responder → returns 127.0.0.1 REFRESH upstream DNS endpoint changes port → resolver updated UNINSTALL → resolver stopped, state file removed
Resources
No Incus resources; in-memory only
Cleanup
Resolver goroutines + temp file

TestDisposableInfrastructureCreateAndDelete

infrastructure_test.go
SANDCASTLE_E2E=1

Creates a full infrastructure project containing a Caddy ingress instance and a route-broker instance. The sandcastle binary is cross-compiled for the remote's architecture and deployed into the route-broker. Verifies mTLS authentication with the broker.

InfrastructureCreator.CreateInfrastructure ├─ create Incus project sc-infra-<runID> ├─ create sc-caddy instance (from base image) ├─ create sc-route-broker instance (from base image) ├─ write runtime files: │ Caddyfile, broker env, broker TLS cert/key, systemd unit ├─ deploy sandcastle binary (linux/<remote-arch>, cross-compiled) ├─ start Caddy (systemctl restart caddy) └─ start sandcastle-route-broker service ASSERT ✓ sc-caddy instance exists ✓ sc-route-broker instance exists mTLS PROBE (Python script inside route-broker) generate self-signed client cert POST https://127.0.0.1:9443/routes (no authorization) ✓ HTTP 401 Unauthorized (mTLS challenge works, cert not trusted)
Cross-compile
buildSandcastleForRemote(t, remote) detects KernelArchitecture via Incus API and cross-compiles GOOS=linux GOARCH=<arch>
Duration
~25 s

TestRouteBrokerAuthorizedMutationE2E

route_broker_mutation_test.go
SANDCASTLE_ROUTE_BROKER_INCUS_SOCKET

Full route mutation test: adds routes, verifies authorization rules (403 for unowned, 400 for DNS failure), port pinning, and removal.

SETUP infra project + Caddy + route-broker (with Incus socket mounted) project1/sandbox1 → route added (201) project2/sandbox2 → route added (201) ASSERT AUTHORIZATION project1 cert tries to delete project2 route → 403 Forbidden add route with unresolvable DNS proof → 400 Bad Request port pinning: second add from same cert on different port → 409 Conflict ASSERT MANAGEMENT list routes for project1 → 200, contains sandbox1 delete project1 route → 200 profile cleaned up + ingress device removed from sandbox1

Cleanup Tests

TestCleanupDisposableResourcesE2E

cleanup_test.go
SANDCASTLE_E2E_RUN_ID required

Standalone cleanup tool. Scans all projects, infrastructure, certificates, image aliases, and local Docker tags for ones matching the given run ID, and deletes them. Skips (not fails) when SANDCASTLE_E2E_RUN_ID is not set. Used to clean up after interrupted runs.

$ SANDCASTLE_E2E=1 SANDCASTLE_E2E_RUN_ID=e2e-20260520-120000 go test -run TestCleanupDisposableResources for each Incus project matching runID → delete (purge) for each infra project matching runID → delete for each certificate matching runID → delete for each image alias matching runID → delete for each local Docker tag matching runID → delete report: "removed 2 projects, 1 infrastructure, 0 certs, 4 aliases, 0 tags"

Coverage Gap Audit

Gaps identified by reviewing each test file against the full feature set. Addressed gaps (✅) were fixed in the same commit that introduced this document. Remaining gaps are prioritised by risk and feasibility.

  • ✅ FIXED
    Storage pool and container profile not validated after project creation

    TestDisposableProjectCreateAndPurge now asserts GetStoragePool() succeeds and the container profile's root device references the per-project pool. Also asserts pool is gone after purge.

  • ✅ FIXED
    Idempotent project creation not tested

    TestDisposableProjectCreateAndPurge now calls CreateProject twice and verifies the second call succeeds.

  • ✅ FIXED
    Volume persistence across sandbox delete/recreate not tested

    TestSandboxLifecycleE2E now asserts all three durable volumes survive sandbox removal, then recreates the sandbox and verifies ingress files are written correctly.

  • HIGH
    Concurrent sandbox creation in same project

    No test issues two CreateSandbox calls simultaneously. IP allocation and volume creation could race. Would need goroutines + WaitGroup.

  • HIGH
    CLI sandcastle list and sandcastle delete not tested

    Only add, enter, and inspect are covered. List and delete are standard user-facing commands with no automated test.

  • MED
    App port conflict detection not tested

    No test tries to assign an out-of-range port or a port already used by another sandbox. PlanSetPort should return an error; no test covers this path.

  • MED
    Restricted user denied sandbox creation in non-granted project

    TestRestrictedUserGrantAccessE2E tests denial of project creation but not sandbox creation in a non-granted project. Those are different authorization checks.

  • MED
    Tailscale re-connection not tested

    Only the initial "up" → running flow is tested. The path of Tailscale going down and recovering (restart sidecar, re-authenticate) has no coverage.

  • MED
    Route broker offline / connection-refused scenarios not tested

    TestDisposableInfrastructureCreateAndDelete only validates the happy path (401 on unauthenticated request). No test exercises the broker being unreachable.

  • LOW
    Certificate renewal / rotation not tested

    Local trust install/uninstall is tested but not reinstall-over-existing or certificate rotation.

  • LOW
    DNS TTL and negative caching not tested

    Tests assert positive/negative responses but do not check TTL values or cache behaviour.

  • LOW
    Storage quota enforcement not tested

    Per-project ZFS pools support quotas but no test validates quota enforcement or exhaustion handling.
