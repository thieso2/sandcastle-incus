Sandcastle OIDC User Guide

Configure tenant-scoped workload identity so Sandcastle machines can use GCP without long-lived cloud credentials.

What OIDC Does

Sandcastle machines do not store GCP keys. Instead, a machine uses a Sandcastle Machine Runtime Secret to request a short-lived OIDC token from the Auth App. GCP Workload Identity Federation exchanges that token for short-lived GCP credentials.

machine -> Sandcastle Auth App -> OIDC JWT -> GCP STS -> GCP access token

Tenant Scoped

Each tenant has its own issuer URL, such as https://auth.example.com/t/thieso2.

Machine Scoped

Tokens include tenant, project, and machine claims so cloud-side bindings can be narrowed.

User Owned

The saved Cloud Identity Config named gcp belongs to the logged-in Sandcastle user.

One-Time GCP Setup

Run this after sc login. It creates or updates the GCP Workload Identity Pool, OIDC provider, service account binding, and saves a Sandcastle Cloud Identity Config named gcp.

sc cloud-identity gcp setup --tenant thieso2

If the Auth Hostname cannot be inferred from your current remote, pass it explicitly:

sc cloud-identity gcp setup \
  --tenant thieso2 \
  --auth-hostname big.thieso2.dev

After setup, the Auth App has a user-owned Cloud Identity Config named gcp. Machines still need that config selected, either explicitly or through a project default.

Set a Project Default

Set the default Cloud Identity Config for a Sandcastle project. This works for project default too.

sc project set-cloud-identity default gcp

With this in place, new machines in that project automatically get workload identity during sc create and during sc c auto-creation.

sc c dev
sc create website/codex

Clear the default when a project should stop injecting workload identity into new machines:

sc project unset-cloud-identity default

Enable Machines

New Machines

If the project default is set, no flag is needed:

sc c test
sc create test

Without a project default, pass the config explicitly:

sc c test --cloud-identity gcp
sc create test --cloud-identity gcp

Existing Machines

For a machine that already exists, run workload enable once. This rotates the Machine Runtime Secret and injects the helper files.

sc workload enable test --cloud-identity gcp

Files Written

PathPurpose
/usr/local/bin/sandcastle-workload-tokenHelper that mints Sandcastle OIDC tokens.
/etc/profile.d/sandcastle-workload-identity.shExports Sandcastle and GCP credential environment variables.
/var/lib/sandcastle/workload/runtime-secretMachine Runtime Secret, owned by the machine Unix user.
/var/lib/sandcastle/workload/gcp-credential.jsonGCP external-account credential file.

Verify in a Machine

Use sc c to run checks inside the machine. Explicit commands run through a login shell, so profile exports are loaded.

sc c test -- 'id -un; env | sort | grep -E "^(GOOGLE|CLOUDSDK)"; gcloud projects list --limit=5'

Expected signals:

  • GOOGLE_APPLICATION_CREDENTIALS=/var/lib/sandcastle/workload/gcp-credential.json
  • CLOUDSDK_AUTH_CREDENTIAL_FILE_OVERRIDE=/var/lib/sandcastle/workload/gcp-credential.json
  • GOOGLE_EXTERNAL_ACCOUNT_ALLOW_EXECUTABLES=1
  • gcloud projects list prints the GCP project.

You can also mint the raw OIDC token:

sc c test -- sandcastle-workload-token

Troubleshooting

No active gcloud account

This usually means no workload files or no GCP profile exports were loaded.

sc c test -- 'ls -la /var/lib/sandcastle/workload; env | grep GOOGLE'

If the directory is missing, enable workload identity or set a project default and recreate the machine.

Machine was created before the project default

Project defaults apply to new machine creation. Existing machines need:

sc workload enable test --cloud-identity gcp

Wrong machine Unix user

New machines use the tenant's stored Unix user. Current CLI versions repair missing tenant Unix-user metadata from the local OS user during sc create and sc c auto-creation. Existing machines keep the user they were created with.

Debug with verbose logs

VERBOSE=1 sc c test -- gcloud projects list --limit=5

Useful log lines include using project default ... cloud identity "gcp", built 9 workload files, and workload identity files: 9.
