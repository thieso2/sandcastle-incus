Why this exists

The hosted-IDE market solved the wrong half of the problem.

Cloud development environments are excellent at provisioning — and indifferent to where your code lives, who holds the keys, and what happens when an AI agent runs unattended for six hours. For a regulated business, that's the entire problem.

Codespaces is fast, until you can't ship your code there.

Hosted platforms put your source, your build artifacts, and your tokens on someone else's infrastructure. For teams with data-residency obligations, regulated customers, or a single contract that says "must remain on company-controlled infrastructure," that's the end of the conversation.

Coder is self-hosted, until you realize you're operating Kubernetes anyway.

DIY self-hosted platforms shift the problem rather than solving it. Now your platform team owns a control plane, a database, a workspace operator, and the on-call rotation that comes with all of it — before you've shipped a single developer environment.

AI agents need what neither of them offer.

Long-running coding agents need persistent environments, real cloud credentials, and meaningful isolation between teams. Handing every agent a service-account JSON file is a credential leak waiting to happen. Sandcastle was built around this constraint, not retrofitted to it.

What you get

Three outcomes that change how your engineering org operates.

Cost & control

A capital-cost line item, not a recurring per-seat bill.

Sandcastle runs on Linux hosts you already operate or rent — bare metal, a hypervisor, or a single VM. There are no per-seat licenses, no consumption meters, and no surprise invoices when your team grows. The cost of giving the fortieth engineer an environment is the same as the first.

Data sovereignty

Source, secrets, and agent traffic never leave your perimeter.

Because Sandcastle runs entirely on your infrastructure, your code is not copied to a vendor's cloud, your secrets are not held in a vendor's KMS, and your AI tools' network traffic does not transit a vendor's egress. You decide what jurisdiction the data lives in, and you can answer "where is our code?" with a specific rack or region.

AI-ready from day one

Codex, Claude Code, Gemini CLI and gcloud preloaded — with credentials that expire.

Every environment ships with the AI coding tools your engineers already use, plus a workload-identity mechanism that mints fifteen-minute OIDC tokens to GCP and AWS. No long-lived service-account keys baked into laptop disks or container images. The agent gets the access it needs, and only for as long as it needs it.

The unique angle

Built for the AI-agent era — not retrofitted to it.

Long-running coding agents have a specific operational shape: they need a persistent workspace, real credentials to talk to cloud APIs, and strong isolation from the rest of the business. Most developer platforms were designed for a human typing at an IDE. Sandcastle was designed for a process that runs for hours without a person watching it.

Persistent, not ephemeral.

Agents need state — checkouts, virtualenvs, dependency caches, partial work. Sandcastle environments live as long as the engineer wants, not for the duration of a single CI run.

Short-lived cloud credentials.

An OIDC provider per deployment issues 15-minute tokens that the agent exchanges for GCP or AWS access. Nothing long-lived ever lands on the box.

Isolation that survives a bad day.

An agent that misbehaves in one tenant cannot touch another tenant's network, DNS, certificates, or storage. Blast radius is the project, not the platform.

Real network identity.

Each tenant is its own Tailnet with its own DNS suffix and private hostnames — so an agent's outbound traffic is auditable and its services are addressable without exposing them publicly.

Security & isolation

What's isolated, and what isn't — in plain language.

We'd rather your security team trust this page than be surprised by an architecture diagram three weeks into the pilot. Here is the honest model.

Per-tenant isolation boundaries

  • Incus project — instances, profiles, and storage volumes are scoped to the tenant's project; a developer in one project cannot see or address resources in another.
  • Network — each tenant runs on its own subnet from a managed CIDR pool, with its own Tailnet membership for cross-machine reachability.
  • DNS suffix — tenants own a private DNS suffix; machine hostnames don't collide and aren't visible across tenants.
  • Certificate authority — every tenant gets its own CA. Leaf certificates inside one tenant are not trusted by another.
  • Storage — workspace volumes are project-scoped. Cross-tenant shares require an explicit grant from the owner and an explicit accept by the consumer.
  • Workload identity — OIDC tokens issued to a machine carry the tenant and machine as claims; cloud IAM bindings can scope access to a specific tenant.

What we don't claim

  • Containers share a host kernel. Sandcastle uses Incus containers by default for speed and density. That is a different threat model from full VMs. Workloads requiring hardware-level isolation should run on Incus VMs, which Sandcastle also supports.
  • The host operator is trusted. A compromised Incus host is a compromised platform. Treat the host like any other Tier-0 production system: minimal access, audited changes, hardened OS.
  • Public routes are public. The shared Caddy broker lets developers expose preview endpoints on your domain. The broker enforces ownership of the hostname, but the endpoint itself is reachable from the internet by design.
  • GitHub is your IdP. Login is gated by a GitHub allowlist controlled by your administrators. If a GitHub account on the allowlist is compromised, the attacker gets the access that account has.

For your staff engineer

How it works, in three commands.

A developer goes from "no environment" to "AI agent running with cloud credentials" in about a minute. This is the part you forward to your staff engineer. 

# 1. Log in. Sandcastle creates a personal tenant on first login.
$ sc login
Opening browser for GitHub OAuth...
Logged in as ada-lovelace.
Created tenant ada with DNS suffix ada.dev.example.com.

# 2. Create a machine from the AI-ready base image.
$ sc machine create dev --image ai
Provisioned dev.ada.dev.example.com.
  network:  ada (Tailnet)
  ca:       ada-ca
  identity: oidc://auth.example.com/tenants/ada/machines/dev

# 3. Connect. From here, agents have GCP access via short-lived tokens.
$ sc connect dev
Connected. Last login: Tue May 27 14:02:11
ada@dev:~$ gcloud storage ls gs://team-data
gs://team-data/2026-Q1/
gs://team-data/2026-Q2/
Self-hosting reality check

You need one Linux host, a domain, and a GitHub OAuth app.

Sandcastle is intentionally boring to operate. It runs on top of Incus — no Kubernetes control plane, no managed Postgres, no Redis, no message bus. State lives in SQLite on the host. Adding capacity is adding a host. Backing it up is backing up a directory. Your platform team will not need to learn a new operations model.

A Linux host (bare metal or VM) with Incus installed.
A domain you control, for the auth host and developer routes.
A GitHub OAuth application for developer login.
A Tailscale tailnet (free tier is fine for evaluation).
No Kubernetes. No Postgres. No managed control plane.

Evaluate it the way real engineering teams evaluate platforms.

Clone the repository, point it at a host, and have a working tenant before lunch. If it doesn't earn its place in your stack, you've spent a morning — not a procurement cycle.
